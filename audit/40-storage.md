# Audit — 40-storage.md

## 1. **How a read transaction learns its generation is unstated, and the "no locks beyond the map's" claim depends on it** [blocker]

The doc says "A read transaction at tx-id *T* gets images for *T*" and "LMDB's MVCC guarantees the builder scanning inside a read txn sees exactly generation *T* — correctness needs no locks beyond the map's." That guarantee only holds if *T* is obtained *from the reader's own snapshot* (a `_meta` get inside the read txn). If *T* instead comes from an in-process atomic/env-scope counter, there is a race: a reader opens its snapshot just before a commit, reads the counter just after, and caches an image built from generation *T* under key *T+1* — silently poisoning the cache for every subsequent reader. This is exactly the quiet-failure class the doc claims to fix, and the stated invariant test ("two sequential read transactions with no intervening write share identical image instances") cannot catch it. Also unstated: who owns the cache (an `Env`-wrapping struct?), and what happens when two readers at the same *T* race to build the same image (both build and one wins? build-once with a per-key latch?).
Question: Is the generation for a read txn defined as the `_meta` storage-tx-id read inside that txn's snapshot, and what is the policy for concurrent duplicate builds — accept wasted double-build, or per-key build coordination?

## 2. **M-namespace hash collision policy is unspecified, and the silent-failure mode is data loss** [blocker]

`M | relation_id | fact_hash -> row_id` is the membership check; the doc never says whether the key is the full 32-byte blake3 or truncated (v5's `fact_handle` at `1b65ae8^:crates/bumbledb-lmdb/src/storage_format.rs` truncated to **16 bytes** — silence here risks inheriting that). If insert treats an M hit as "fact exists" without verifying, a collision makes `insert(new_fact)` a silent idempotent no-op — the exact silent-truncation/skip door this design is supposed to close. Note the inconsistent rigor: the dictionary's forward hash map explicitly does "equality verification on lookup," but M — the same hash-keyed pattern — states no verification; and for `_dict` itself, verification with no stated recovery path is incoherent (if the forward key is occupied by a different string, the new string *cannot be interned* — detected collision has no handling but a panic, which should be said).
Question: Is M's key the full 32-byte hash, does an M hit get verified against `F` (row_id → fact_bytes compare) before being trusted, and is a detected collision in M or `_dict` a hard error?

## 3. **The delete path is entirely unspecified** [blocker]

The "Write path" section describes only insert ("Per inserted fact: one `F` put, one `M` put, guard puts…"), yet 10-data-model makes delete half the mutation algebra ("There is no update operation. Mutation is delete + insert."). Unanswered: delete presumably resolves `fact → M → row_id → F del + M del`, reconstructs `U` guard keys by slicing the constrained fields out of fact_bytes, and deletes its own `R` entries (`target_rel | constraint | key | source_rel | source_row` — all reconstructable from the fact plus its row_id); none of this is written down, and the `U` value being only a row_id means guard cleanup *requires* re-deriving keys from the fact — an implementer could instead guess a scan. Restrict-FK enforcement on deleting a *target* fact (an `R` prefix scan over `target_rel | constraint | key`) is also nowhere stated, and it interacts with the OPEN FK-timing item. The doc's own rule is that code deciding silently is the failure mode.
Question: Write the delete path with the same specificity as insert — which gets/dels against F/M/U/R per deleted fact, and how is Restrict checked on target-fact delete?

## 4. **"Stale-generation images fall out by key" is not a mechanism — the cache as described leaks every generation** [design-gap]

A map keyed by `(relation_id, field_scope, storage_tx_id)` does not shed old entries when the tx id bumps; keys "falling out" requires an explicit eviction action the doc never specifies (drop-on-commit? drop when no reader at that generation remains?). Combined with `field_scope` in the key, retained `Arc`s per scope per generation grow without bound in any long-running process with writes. Eager drop-on-commit has its own hazard the doc must address: a still-open reader at generation *T−1* loses the shared cache and silently degrades to per-query rebuilds — v5's exact quiet failure, back through a side door. And the scale axiom implies no memory-pressure eviction, but the doc should say that explicitly rather than leave it to be guessed.
Question: What precisely removes an image from the cache, what happens to readers still pinned at older generations, and is "no memory-pressure eviction, ever" the stated policy?

## 5. **`field_scope` in the cache key contradicts both the "tiny key space" claim and 30-execution D1** [design-gap]

The cache is "keyed by `(relation_id, field_scope, storage_tx_id)`" while the same section justifies caching only unfiltered images by "keeping the cache key space tiny and hit rates high" — but field scopes are combinatorial (2^fields per relation), and two queries touching overlapping column subsets build two images that duplicate columns in memory. 30-execution D1 says images are "built once per (relation, tx-generation)" — no field_scope — so the two docs disagree about the cache identity. The obvious alternative (always build all columns of a relation; at ≤100s of MB the whole relation is small) is not recorded, violating the README's decision-plus-alternative rule.
Question: Is the image unit per-field-scope or all-columns-per-relation, and if per-scope, why did all-columns lose given that the scale axiom makes full images cheap?

## 6. **Storage tx id semantics: "every committed logical write advances" vs "flushed once at commit"** [design-gap]

One paragraph says counters including the "storage tx id" are "maintained in memory during the write transaction and flushed **once at commit**"; the next says "Every committed logical write advances the **storage tx id**." If a txn does 100 inserts, does the id advance by 100 or by 1? It only matters at the edges, which are unaddressed: a write transaction whose logical writes are all idempotent no-ops (set semantics — inserts of existing facts) makes no state change; if it still bumps the id it needlessly invalidates every cached image. Since the id lives in `_meta` and commits atomically with data, crash consistency is fine — but the bump granularity and the no-op-commit rule need one deciding sentence.
Question: Does the tx id advance once per commit that changed state, per commit unconditionally, or per logical write — and does a commit of pure no-op writes invalidate the image cache?

## 7. **COLT offsets vs non-dense row_ids: the mapping is never stated** [design-gap]

The paper's COLT leaves are "a vector of offsets into the base relation" where "each column is stored as a vector" — dense positional indexes (tex/04-optimizations.tex §4.2). Bumbledb's row_ids are "monotonic" per relation, and deletes leave holes, so row_id ≠ image position; the doc must say that image positions are the dense scan ordinals of the F-prefix scan (0..row_count) and that row_ids never appear inside images — or an implementer will index columns by row_id and either waste memory on holes or read garbage. Relatedly, anything that bridges from M/U point lookups (which yield a row_id) into image-based execution needs a row_id→position story, or an explicit statement that point lookups answer from F directly and never touch images.
Question: State the invariant — COLT offsets are positions in the image's scan order, row_ids exist only in LMDB keys — and say whether any path ever needs row_id→image-position translation.

## 8. **The S namespace is a placeholder, but 30-execution's planner needs specific statistics** [design-gap]

`S | relation_id | stat -> value` is glossed as "row counts, stats" — a definition containing itself. 30-execution requires "per-relation row counts (maintained on write, exact); per-filter survivor estimates from cheap heuristics; unique/FK constraint knowledge" and declares "a statistic that isn't real doesn't exist in the struct." Constraint knowledge comes from the schema, and row counts are named here — but if per-filter survivor heuristics need any durable inputs (per-column min/max, distinct-ish counts), S is where they'd live and nothing defines them; if they need none, S is just row counts and should say so. As written, an implementer invents stat keys, which is the fake-statistics door v5 walked through.
Question: Enumerate the exact contents of S (keys and values) and confirm the planner's survivor heuristics need nothing durable beyond row counts.

## 9. **Serial (Q) allocation mid-transaction is not covered by the counter-flush story** [design-gap]

The counters "maintained in memory during the write transaction and flushed once at commit" are enumerated as "(`row_id` high-water, row counts, storage tx id)" — serial sequences are conspicuously absent, yet `Q` values have the same requirement: allocating two serials in one txn must see the txn's own prior allocation (read Q once at first use, increment in memory, flush at commit), and 10-data-model's "Aborted transactions don't advance the committed sequence" falls out only from that discipline. 10-data-model also allows ETL to "supply explicit values, which advance the high-water mark" — interleaving explicit and generated values in one txn needs a stated max-tracking rule (an explicit value below the in-memory next must not regress it; a generated value after a high explicit one must skip past it).
Question: Confirm Q counters join the in-memory-then-flush set, and specify the mixed explicit/generated allocation rule within one transaction.

## 10. **Corrupt-data behavior at image build is unspecified — the old silent-skip door is not yet closed** [design-gap]

The post-mortem lineage this doc invokes includes code that silently truncated values and silently skipped corrupt rows, yet nothing here states what the image builder does when an `F` value's length does not equal the schema's fixed fact width, or when a decoded intern id has no `_dict` reverse entry, or when M/F/U disagree (guards are derived data; a bug can desync them). "Parse, don't validate" implies these are hard errors that fail the scan, but that must be written — the difference between "hard error" and "skip and continue" is invisible in the happy path and is precisely how v5's corruption handling went quiet. There is also no integrity-check story (even a debug-build M↔F↔U consistency sweep) and no statement that one is out of scope.
Question: State the policy — any structural inconsistency encountered during scan/decode (wrong fact width, dangling intern id, M/F mismatch) is a hard error, never a skip — and say whether an offline consistency checker is in or out of scope.

## 11. **The single `_data` database is an unrecorded decision, and it quietly breaks the append-mode claim** [design-gap]

"one `_data` database, first-byte namespaces" appears as a heading, not a decision — no alternative recorded, violating README rule 1. The alternative (one LMDB database per namespace) is strong: it enables `MDB_INTEGERKEY`-style layouts, per-namespace stats, and — decisively — append mode. In a single keyspace, `F` (0x46) sorts before `M` (0x4D), `Q`, `R`, `S`, `U`; LMDB's append mode requires each put's key to exceed *every existing key in the database*, so once any M/Q/S entry exists, no `F` put can ever append. The doc's "Bulk load sorts by key and uses LMDB append mode where key order allows" is therefore true only for a globally sorted load into a *fresh* database with all namespaces buffered and written in namespace order — a real constraint an implementer must know, not discover.
Question: Record the one-database-vs-per-namespace decision with its alternative, and state that append mode applies only to fresh-database bulk loads written in global key order.

## 12. **Cold-build filter pushdown contradicts "only unfiltered images are cached"** [design-gap]

"Filters push down into the image build when an image is being built anyway (scan, test, keep survivors) — but filtered *views* are query-local; only unfiltered images are cached." If the cold build keeps only survivors, the thing built is filtered and by the second clause cannot be cached — so the first filtered query against a cold relation either populates nothing (next query rebuilds; v5's rebuild disease in miniature) or the build must do double duty, emitting the full unfiltered image for the cache *and* a survivor view for the query in one scan. The doc doesn't say which, and 30-execution D1 describes only cached unfiltered images, so the pushdown sentence has no cross-doc anchor either.
Question: On a cold build with a filter present, does one scan produce both the cached unfiltered image and the query-local survivor view — and is a filtered view a survivor-position vector over the image (COLT-compatible offsets) or a compacted copy?

## 13. **The doc's design inputs cite a post-mortem that does not exist in git history** [clarification]

The layout is justified by "(post-mortem `todo/20`–`todo/24` at `1b65ae8^`)", and README points to "the full post-mortem review … in `todo/` at `1b65ae8^`" — but no `todo/` path exists in any of the repository's 292 commits (verified by walking every tree; `1b65ae8^` = 465e3d4 contains only crates/docs/fuzz/scripts). The review dump was evidently never committed and was deleted in the wipe. The quantitative claims it backs ("stored every field value 4–6 times… ~15 B-tree puts per inserted fact… 80% of traced execution") are now unverifiable by anyone but the author, in a doc whose stated purpose is to encode those lessons; v5's actual namespace roster is still recoverable from code (e.g. `1b65ae8^:crates/bumbledb-lmdb/src/storage_format.rs`), so the citation could at least point there.
Question: Recover and commit the post-mortem (or re-point the citations at the surviving v5 source paths), so the design inputs are auditable.

## 14. **Open-time verification and id widths are split across docs and half-unstated** [clarification]

`_meta` holds "format version, schema fingerprint, storage tx id," but only the fingerprint check is specified — and in 10-data-model ("Open compares fingerprints; mismatch is a hard failure"), not here. What the *format version* check does on mismatch (hard fail, same as fingerprint?), and when either bumps, is unstated; so are the widths and assignment of `relation_id`, `field_id`, and `constraint` in keys (v5 used u32 relation ids from schema declaration order — presumably the fingerprint pins the assignment, but say so). These are exactly the values an implementer guesses at on day one.
Question: State here that open verifies format version then fingerprint (both hard failures), and pin the key-component widths and the rule that ids come from canonical schema order covered by the fingerprint.

## 15. **Backup, compaction, and file-growth are absent — in scope or out?** [clarification]

LMDB data files never shrink; deleted pages recycle but the file only grows, the dictionary leaks by accepted design (10-data-model), and single-writer means a hot copy needs `mdb_copy`-style support or a quiesced writer. The non-goals list in 00-product covers migrations ("ETL into a new database") but says nothing about backup or reclaiming space in a long-lived database, and this doc — the storage authority — is silent. Given the scale axiom this may all be a one-line non-goal ("copy the file; ETL is compaction"), but per README rule 3, unwritten is the failure mode.
Question: Add an explicit operational paragraph — backup = file copy / `mdb_copy`, compaction = ETL into a fresh database, nothing else exists — or mark it OPEN.
