# Audit: 50-validation.md

## 1. **The SQLite aggregate oracle is unsound as specified — SUM over joined rows is not SUM over the binding set** [blocker]

The doc says "Every benchmark and golden query is executed against SQLite with `SELECT DISTINCT`" and the ledger suite includes "balance-style aggregates by account and instrument (when aggregate execution lands)." But `SELECT DISTINCT` and `GROUP BY` do not compose into bumbledb's semantics: bumbledb folds "over sets of bindings" (20-query-ir), while SQLite's `SELECT account, SUM(amount) ... GROUP BY account` folds over the *bag* of joined rows — a many-to-many join (PostingTag, OrgParent hops) inflates the sum by witness multiplicity. `DISTINCT` applied to the *output* deduplicates result rows (too late), and `SUM(DISTINCT amount)` deduplicates *values* (wrong — two distinct bindings with equal amounts must both count). The only faithful oracle form is `SUM` over a `SELECT DISTINCT <all binding variables>` subquery, and `Count` (which "counts the group's binding set, exactly") becomes `COUNT(*)` over that same subquery, not `COUNT(x)` or `COUNT(DISTINCT x)`. This is precisely the "count-equality masked real bugs" failure mode reborn one level up: an oracle that agrees on easy queries and silently diverges on the ones the product thesis is about.
Question: What is the exact SQL template for each aggregate op (Sum/Min/Max/Count) — specifically, is the oracle defined as the aggregate over a `SELECT DISTINCT`-of-all-binding-variables subquery, and is that written into this doc before aggregate execution lands?

## 2. **"Exactly, by value" is asserted but the value mapping between bumbledb and SQLite is never defined — the old oracle silently coerced everything to `Serial(parse().unwrap_or(0))`** [blocker]

The doc's core claim — results "must equal SQLite's **exactly, by value**" — has no definition of what a value *is* on the comparison boundary. Interned Strings/Bytes are 8-byte dictionary ids in facts; the ids are database-local and meaningless to SQLite, so comparison must decode to raw bytes — where? Enums are 1-byte ordinals of a schema-declared closed domain: compared as the ordinal integer or the variant name TEXT? Serials are *nominal* (`AccountId` ≠ `InstrumentId`) but SQLite has one INTEGER; does the harness carry the type tag on the bumbledb side and strip it, and can a mapping bug conflate columns? Bytes must map to BLOB, not TEXT (SQLite `DISTINCT` distinguishes `X'41'` from `'A'`). This is not hypothetical: the v5 oracle (`1b65ae8^:crates/bumbledb-bench/src/job/sqlite.rs`) shelled out to the `sqlite3` CLI, split stdout on `|` (breaking on any string containing a pipe), and parsed every cell as `Value::Serial(value.parse().unwrap_or(0))` — an "exact comparison" that silently mapped all strings and parse failures to 0. Nothing in this doc prevents rebuilding exactly that.
Question: Specify the normative column-type mapping (each bumbledb type → SQLite storage class), the comparison representation (typed API values via rusqlite, never CLI text), and where string/bytes intern-id → raw-bytes decode happens — which side of the timed region and which side of the comparison?

## 3. **U64 values above i64::MAX do not fit SQLite INTEGER — no comparison or generation rule exists** [blocker]

Bumbledb has `U64`; SQLite INTEGER is a signed 64-bit value. A U64 above `i64::MAX` inserted through any typed SQLite API either fails, silently becomes REAL (loses precision — and 00-product forbids floats), or becomes TEXT (changing join/DISTINCT/comparison semantics on the oracle side). The doc specifies neither a mapping (e.g., order-preserving 8-byte BLOB? decimal TEXT? reject?) nor a constraint on the data generator ("seeded and reproducible" says nothing about staying under 2^63). A differential fuzzer that never generates a high-bit U64 proves nothing; one that does, with no defined rule, produces unexplainable oracle disagreements. Serial high-water values fed by ETL "may supply explicit values" (10-data-model), so 2^63+ is reachable in principle, not just adversarial.
Question: What is the rule — U64 constrained to < 2^63 in all oracle-checked data (and enforced where?), or an explicit encoding into SQLite with defined comparison semantics for range predicates?

## 4. **"Beats SQLite on the ledger benchmark" has no specified SQLite methodology — the headline success criterion is currently unfalsifiable-honest** [blocker]

00-product criterion 2 names this doc as the ratchet, but the doc says nothing about how SQLite is configured or timed: in-memory (`:memory:`) vs file-backed; WAL/journal mode; page cache size; which indexes are created (bumbledb gets unique guards and an optimized engine — an unindexed SQLite is a strawman; a fully-indexed one is the honest opponent); prepared statements reused vs re-parsed per run; `ANALYZE` run or not; warm vs cold; what is timed (execution only vs including result materialization — bumbledb's aggregate sink "never materializes the join," so the timed regions must be defined to be comparable); repeats and the statistic (the paper reports geometric means, excludes selection/aggregation time, single-threaded, default DuckDB config — §5 of `tex/05-eval.tex` is the honesty bar and it states every one of these). The old runner timed bumbledb against pre-computed "embedded exact expected rows" and never timed SQLite at all in the quick preset, so there is no inherited methodology to fall back on.
Question: Write the SQLite comparison protocol into this doc — storage mode, indexes (and the rule for choosing them), statement preparation, warmup, repeat count, statistic, and the exact timed region on both sides.

## 5. **NULL arises on the SQLite side even though bumbledb has none — empty-input aggregates need a comparison rule** [design-gap]

Bumbledb: "Empty groups do not appear (no input rows → no group → no null, ever)" (20-query-ir). SQLite agrees for `GROUP BY` (no rows → no groups), but an *ungrouped* aggregate over empty input returns one row: `SELECT SUM(x) FROM t WHERE 1=0` → a single NULL (`COUNT` → 0). If any golden/random query aggregates with all output fields aggregated (no group key), the oracle emits `[NULL]` where bumbledb emits `{}` — the harness must define whether that's a translation rule (SQL template always has a group key? wrap in a NULL-filtering subquery?) or a hard harness error. Left unspecified, an implementer will "fix" it ad hoc in the comparison loop, which is where the last oracle rotted.
Question: Are zero-group-key aggregate queries representable in the IR, and if so, what is the defined SQLite oracle form and NULL-handling rule for them?

## 6. **Fuzzing vanished without a decision — the old repo had fuzz targets and the "deliberately do not have" list doesn't mention them** [design-gap]

The pre-reset tree had a fuzz crate with `fuzz_encoding_decode.rs` and `fuzz_query_executor_differential.rs`, and the old validation contract listed "Fuzz crate check" as required. This doc's "What we deliberately do not have" enumerates rejected gates (line-count, PRD-map, coverage %, …) but fuzzing appears nowhere — neither kept nor rejected, which per the README's own rule ("the failure mode is code deciding it silently") is exactly the bad state. An encoding round-trip fuzzer is the natural guard for the order-preserving encodings (sign-flipped i64, big-endian u64, composite guard keys in `40-storage.md`) where a boundary bug (i64::MIN, empty bytes, max-length interned values) corrupts sort order silently. The "randomized queries" bullet partially covers the executor-differential target but not the encoding one.
Question: Is dropping fuzzing deliberate (then say so and why in the do-not-have list) or is coverage-guided fuzzing of encoding round-trips and IR validation retained?

## 7. **No crash/abort/reopen test family — "LMDB atomicity is the whole crash-consistency story" is an untested claim** [design-gap]

40-storage commits counters (row_id high-water, row counts, storage tx id, serial sequences) "in memory during the write transaction and flushed **once at commit**," and the write path does multiple puts per fact across F/M/U/R/dict namespaces. The old contract required "Failpoint tests around dictionary, access, canonical fact, stats, and commit stages"; this doc has abort coverage only inside operation-sequence property tests, and nothing reopens the environment. A deferred-counter-flush bug (serial high-water never persisted; row count drifting from F-namespace truth; stats a query planner trusts as "exact") is invisible to any test that never closes and reopens the database — and a kill-during-commit test is the only way to check that the claim "Failed transactions leave nothing" survives the actual write-path code rather than just LMDB's marketing.
Question: Is there a reopen-after-commit / kill-and-recover test family (process abort between logical steps, then reopen and assert F/M/U/R/Q/S mutual consistency), and if failpoints are rejected as v5 ceremony, what replaces them?

## 8. **No reader/writer concurrency tests — the image cache's trickiest states are all concurrent, and the old repo's "quietest failure" was an image-cache bug** [design-gap]

The image-cache invariants tested here are sequential: "identical instances across read txns at the same tx id; invalidation after every commit." But the cache is environment-scoped, shared across threads, keyed by `(relation_id, field_scope, storage_tx_id)` (40-storage) — its hard cases are concurrent: a long-lived reader at generation T while the writer commits T+1, T+2 (do T's images survive for that reader, or does "stale-generation images fall out by key" evict what a live reader needs — and either way, who tests it?); two readers racing to build the same missing image; rapid write/read interleaving where a reader must never see a T-1 image at T. v5's silently-per-transaction cache — "the old repo's quietest failure" per 40-storage — was exactly a sharing bug that sequential same-generation tests can be written to pass while the concurrent behavior is wrong. Also untested: writer-advances-tx-id vs reader-opens-snapshot ordering (a torn read of the tx id yields a wrong cache key with a correct snapshot, which is a wrong-results bug, not a perf bug).
Question: What test family exercises concurrent readers + the single writer against the image cache (staleness, eviction under a pinned old reader, build races), and is loom/threaded stress in scope or rejected?

## 9. **ETL/bulk load has no test family, yet ETL is the schema-migration story** [design-gap]

00-product: "Migrations (ETL into a new database is the schema-change story)." 40-storage: "Bulk load sorts by key and uses LMDB append mode where key order allows." 10-data-model: ETL "may supply explicit [serial] values, which advance the high-water mark." None of this appears in 50-validation: no bulk-load ≡ sequential-insert equivalence test, no test that append-mode ordering violations are rejected rather than corrupting (LMDB append mode with unsorted keys is a classic silent-corruption source), no test that explicit serial values interleaved with generated ones never collide after the high-water advance, no whole-database ETL round-trip (dump → new schema → reload → oracle-equal results). If ETL is the migration story, an ETL bug is a data-loss bug, and it currently has zero coverage.
Question: What is the ETL correctness family — at minimum bulk-load/sequential equivalence, explicit-serial high-water property tests, and an end-to-end ETL round-trip checked against the oracle?

## 10. **"Zero steady-state allocation" is a boolean over an undefined region — warmup, parameter variation, and intervening writes all legitimately allocate** [design-gap]

The gate "asserts the prepared-query zero-steady-state-allocation contract... on representative ledger queries," but steady state is undefined. First execution builds columnar images (allocates, sanctioned via D1), forces COLT nodes into arenas, populates the plan cache, and touches lazy statics and hashmap growth in global state; a *write* between executions invalidates images so the next execution rebuilds (allocates again); a different parameter binding can force previously-unforced COLT subtries — is arena *growth* (new chunk malloc) an allocation or amortized-and-exempt? The result buffer is "the single sanctioned allocation site" — how does the counting allocator distinguish it (caller-provided buffer sidesteps this; the convenience path doesn't)? And a global counting allocator counts every thread in the process, so the test protocol must be single-threaded or thread-attributed or it flakes. Without these definitions the boolean is implemented by whoever writes the harness, and the chronic-disease detector becomes exactly as trustworthy as an unreviewed test fixture.
Question: Define the gate protocol: N warmup executions of the prepared query, then M measured executions with (a) parameters fixed or drawn from a fixed set? (b) no intervening writes? (c) arena growth counted or exempt? (d) counter scoped how (single-threaded harness / thread-local)?

## 11. **Random query generation has no coverage requirements — the semantics-critical IR features won't arise by chance** [design-gap]

"Randomized queries over randomized ledger-shaped data" is the bug-finder, but nothing requires the generator to hit the features where bumbledb's semantics actually diverge from naive intuition: repeated variables within one atom (same-fact equality, 20-query-ir), existential-only variables triggering D2's early-subtree-skip (an *optimization unique to this engine* — precisely where a set-semantics bug would hide, and unreachable unless the generator deliberately creates high-multiplicity existential suffixes), `Param` terms exercising prepared-query re-execution with varying bindings (plan reuse bugs), empty relations, comparisons on every orderable type, unique-key point lookups, and the cyclic query. Unsteered random generation over ledger-shaped schemas will produce mostly acyclic FK-walk queries and quietly never test the interesting 20%. The old repo's differential fuzz target existed and still missed the bugs that mattered, which suggests coverage, not existence, was the gap.
Question: Does the query generator have an explicit feature-coverage contract (each IR construct provably generated, D2-skip paths exercised with duplicate-witness data), and is that contract itself asserted?

## 12. **The oracle requires an IR→SQL translator — untested infrastructure sitting between the engine and the "oracle"** [design-gap]

"SQLite appears only as an oracle; it is never infrastructure" — but randomized differential testing means every random IR query must be compiled to SQL by harness code. That translator is nontrivial (named-field bindings → column lists, repeated in-atom variables → self-equality predicates, existentials → omitted from SELECT DISTINCT, the aggregate subquery form from finding 1, intern-id → literal decoding) and a bug in it makes every 3-way disagreement ambiguous: is the engine wrong, the reference engine wrong, or the SQL wrong? The old repo dodged this with hand-written SQL per golden query (and still got the *comparison* wrong, finding 2). The doc should name the translator, its testing story (e.g., hand-written SQL goldens that pin the translator's output for known queries), and the arbitration rule when engine, reference, and SQLite disagree pairwise.
Question: Who owns and tests the IR→SQL lowering, and what is the documented arbitration procedure for a 3-way disagreement?

## 13. **The ratchet's content is a moving target across the aggregate-phasing OPEN — "beats SQLite" is measured against an unfrozen suite** [design-gap]

The ledger suite includes aggregates "when aggregate execution lands," and README marks aggregate phasing OPEN. So success criterion 2 can be claimed on the join-only suite, then the suite silently changes shape when aggregates land — either the claim is retroactively hollow or aggregates arrive under pressure to not regress a headline number. The same ambiguity applies to criterion 1's "full validation suite." A ratchet whose membership is implicit is v5's "benchmark quietly *became* the product thesis" risk inverted: the thesis quietly excused from the benchmark.
Question: Is the ratchet suite an explicit, versioned list of named queries (with the rule that the beats-SQLite claim is void until the aggregate families are in it), and where does that list live?

## 14. **Validation-boundary rejection has no differential story — SQLite accepts what bumbledb rejects** [clarification]

20-query-ir rejects typed errors at the boundary: cross-serial unification, comparisons over non-orderable types (string ordering unsupported per 10-data-model), aggregates over unbound variables. SQLite happily compares TEXT, orders anything, and unifies all integers — so the oracle is structurally silent on rejection behavior, and the random generator must either never emit invalid IR (losing negative coverage) or the harness must assert typed-error identity without an oracle. Enum ordering is a concrete instance: `Min/Max over the value types' total order` (20-query-ir) — if Enum is orderable by ordinal, the SQLite mapping (finding 2) must order identically (ordinal INTEGER, not variant-name TEXT) or Min/Max golden queries diverge.
Question: Is there a negative-testing family for boundary rejection (invalid-IR corpus with pinned error kinds), and is Enum orderable — and if so, ordered identically on the oracle side?

## 15. **"Golden query" is referenced but never defined** [clarification]

The doc says "Every benchmark and golden query is executed against SQLite" — the benchmark queries are enumerated by family, but the golden set is otherwise unmentioned. The old contract had an explicit golden roster (ledger, sailors, joinstress, TPC-H/JOB/Lahman/LDBC subsets) covering "duplicate witnesses, exact projection sets, duplicate insert no-ops, absent delete no-ops, constraint behavior." Presumably most of that roster died with JOB-chasing, but *some* permanent hand-written fixtures with known answers are the anchor when the random 3-way differential (finding 12) disagrees, and their scope is currently a guess for the implementer.
Question: What is the golden set — hand-written queries with hand-verified expected results over a fixed dataset, and which of the old roster's bug classes (duplicate witnesses especially) must it cover?

## 16. **Performance-regression classes with correct results have no detector — wrong cover choices and silent scalar fallback pass every test here** [clarification]

v5's two flagship engine failures were performance bugs with correct output: cover selection "systematically chose wrong covers" by comparing inflated estimates to exact counts (30-execution), and the "vectorized" mode that ran scalar per tuple. Scalar/vectorized *equality* testing catches wrongness, not fakeness; the oracle catches nothing here; only the benchmark could — and a ~2x regression on a suite that still "beats SQLite" ratchets nothing. `EXPLAIN` exposes "per-node estimated vs actual cardinalities + cover choices" (30-execution), so cheap assertions exist in principle (e.g., a fixture where the correct cover is forced by construction, asserted via EXPLAIN). This may be deliberately out of scope for a gate-averse project — but then the doc should say the benchmark is the *only* performance detector and accept that class.
Question: Do any EXPLAIN-based assertions (cover choice on constructed skew fixtures, vectorized-path-taken) exist in the suite, or is the ledger benchmark's timing knowingly the sole detector for correct-but-slow regressions?
