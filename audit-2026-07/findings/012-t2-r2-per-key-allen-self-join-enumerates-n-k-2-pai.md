## t2/r2: per-key Allen self-join enumerates Σ n_k² pairs; ~2.7% survive — the order-based overlap join is absent

category: perf | severity: high | verdict: CONFIRMED | finder: perf:olap-temporal
outcome: fixed be405715

### Summary

`t2_overlap_join` — the pairwise span-overlap self-join per key, the slowest temporal lane at 162.8ms night-run p50 — executes as a generic free join: the leaf enumerates every (a,b) span pair within each key (Σ n_k² ≈ 20.6M pairs over 150,034 spans / 2,000 keys), and only *after* materializing each batch do the `a<b` residual and the Allen INTERSECTS residual filter it down to 278,768 survivors (~2.7% selectivity — ~74x more enumeration/classify work than output). Because spans per key can be start-sorted, a two-pointer/active-set sweep per key computes the identical pair set in O(n_k log n_k + out). That machinery family already exists once in the repo (`interval/sweep.rs`, owned by Pack's finalize and the containment checker) but is never applied to join enumeration. The same residual path prices `r2_temporal_ring` (62.9ms; ratios 0.42/0.60, the worst non-bomb ratios alongside r1's 0.56).

### Evidence (all verified against the code)

- **Query shape**: `crates/bumbledb-bench/src/scenarios/temporal.rs:155-182` — two `Span` atoms joined on the key var, with `lt(var(0), var(1))` and `allen(var(3), var(4), AllenMask::INTERSECTS)` as conditions.
- **Enumeration**: `crates/bumbledb/src/exec/run/run_node.rs:143-162` — `iter_batch` over the cover's per-key cursor yields every position; nothing about the enumeration is interval-aware.
- **Post-hoc filtering**: `run_node.rs:278-301` (the `a<b` whole-value residual) and `run_node.rs:333-369` (the Allen residual: gather four endpoint streams per survivor, `allen_code_batch` classify, `allen_filter_batch` mask, compact). Every pair is materialized into scratch before either filter runs.
- **The lever is absent by design, and recorded as such**:
  - `crates/bumbledb/src/ir/normalize.rs:253` — "Interval-pair predicates are NEVER decomposed — they are `PlacedAllen` residuals carrying their mask."
  - `crates/bumbledb/src/exec/kernel/allen.rs:88-96` — the NOTE records only the *kernel-level* lever (bind-time two-compare INTERSECTS monomorphization, "a recorded lever, not shipped"); no asymptotic/enumeration lever is recorded anywhere.
  - Grep of `exec/` and `plan/` for sort-merge/sweep/overlap-join machinery: the only sweep users are `exec/sink/aggregate/finalize.rs` (Pack) and the containment checker — exactly the two `Continuation` owners of `crates/bumbledb/src/interval/sweep.rs:59-111`.
- **Bench pins** (`bench-out/night-2026-07-20/scenarios/scenarios.md`): `t2_overlap_join` ours 162794.7us, sqlite DNF>1000ms, sqlite-tuned 509756.0us (ratio 0.32); `r2_temporal_ring` ours 62857.3us, ratios 0.42 (sqlite) / 0.60 (sqlite-tuned); `r1_wash_ring` 0.56 — r1/r2 are the worst non-bomb ratios in the run.
- **Corpus arithmetic independently reproduces the finder's counters** (`crates/bumbledb-bench/src/scenarios/temporal/corpus.rs`): 150,000 random + 34 planted rows = 150,034; key 0 takes the 1-in-50 Zipf redirect → ~3,078 spans → ~9.5M of ~20.4M total per-key pairs; expected INTERSECTS selectivity from the span geometry (widths ≤ 2 days over a ~347-day horizon, ~2% rays) computes to ~2.6-2.7%, matching 278,768 survivors of ~10.3M classified. The claimed introspection counters (`pass=10554781 fail=20423292`, 30.98M total residual evals) are exactly self-consistent with P ≈ 20.65M pairs through the two-residual pipeline (P `a<b` evals + P/2 Allen evals; pass ≈ P/2 + 278,768). One correction: the Allen classify count is ~10.28M, not the summary's 10.55M — the finder conflated the combined pass counter; immaterial.
- **Spec check**: the Free Join paper (`docs/free-join-paper/arXiv-2301.10841v2/tex/`) contains no interval/inequality/theta-join machinery — the Allen residual path is the repo's own extension, so the paper neither mandates nor excuses the all-pairs shape. Crucially, `docs/architecture/40-execution.md:85-97` already declares "interval-overlap joins are O(n) (image scan + filter) in v0 — decided", policed by the latency budget with the range-accelerator OPEN item armed "on violation"; `docs/architecture/00-product.md:85-98` sets that budget at p99 ≤ 10ms (informational until scale L exists). At bench scale — well *below* the 10⁷-fact ceiling — t2 sits 16x past the informational budget. This finding is the recorded trigger's bench-scale evidence, not the discovery of an unknown gap.

### Bench impact

The enumeration is the lane: ~20.6M pair materializations + ~20.6M `a<b` compares + ~10.3M Allen classifies for 278,768 answers. A per-key start-sorted sweep does Σ n_k log n_k ≈ 1-1.5M comparisons plus ~279K emissions — and because any interval still active at b's start already intersects b (a.start ≤ b.start < a.end, and a.start < b.end trivially), the sweep emits only true intersecting pairs, shrinking the classify kernel's input from 10.3M to ~279K. t2 drops from ~163ms toward the ~5-15ms class; r2_temporal_ring's Allen phase shrinks proportionally, lifting the run's two worst non-bomb ratios (0.42/0.60).

### Suggested fix

Represent connected Allen masks (INTERSECTS and supersets — detectable from the mask bits at bind time, the same bind-time detection the allen.rs:88-96 NOTE already sketches for the two-compare lever) as a per-key sort-merge/sweep enumeration when both interval variables are covered at the same leaf under a shared key prefix: force the per-key suffix once, sort positions by start word (or maintain a start-sorted per-interval-column suffix representation), and emit pairs by frontier walk — the `interval/sweep.rs` Continuation pattern extended with an active set. The mask stays data: sweep-emitted candidate pairs still flow through the uniform `allen_code_batch`/`allen_filter_batch` kernels, so only the *enumeration* changes representation — consistent with the design doctrine (representation over control flow) and with the architecture doc's own armed OPEN item, which this would discharge.
