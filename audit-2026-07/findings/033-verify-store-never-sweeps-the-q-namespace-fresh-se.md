## verify_store never sweeps the Q fresh-sequence namespace: the never-reissue high-water is unverified offline

category: missing-free-feature | severity: medium | verdict: CONFIRMED | finder: engine:storage

### Summary

The offline sweeper `Db::verify_store` walks every namespace of the `50-storage.md` key layout — F facts, M membership, U determinants, R reverse edges, the window-marks roster, S counters, the `_meta` descriptor, and the `_dict` leak statistic — but has no pass over `Q | relation_id | field_id -> next_u64`, the fresh-sequence namespace. Q carries exactly the same monotone-high-water law the sweeper already convicts for S row ids (`RowIdHighWaterLow`): every committed fresh-field value must sit below the stored Q next-value, or `alloc()` re-issues an id the host already holds — the law the entire burn-on-abort machinery (`flush_escaped_fresh_ids`, `EscapedIdBurn`) exists to uphold, proved as `lean/Bumbledb/Txn/Fresh.lean:283 never_reissue_observable`. A desynced Q entry, a malformed Q key, or an alien-relation Q key all pass the sweep silently, while every other namespace's equivalent is a finding.

### Evidence (all verified against the code)

- `crates/bumbledb/src/verify_store.rs:259-264` — the complete pass roster: `facts::sweep, membership::sweep, determinants::sweep, reverse::sweep, marks::sweep, counters::sweep`, then the descriptor check (270-280) and `dict_stat::dangling` (281). `grep -rn "NS_FRESH\|fresh_key" crates/bumbledb/src/verify_store*` returns nothing. The module-doc pass table (verify_store.rs:10-26) lists F/M/U/R/marks/S/_meta/_dict — no Q line, and unlike the dict leak ("the accepted leak ... never a finding", verify_store.rs:85-88) there is no documented acceptance of the gap anywhere; the one "asymmetry is chosen, not overlooked" paragraph in `docs/architecture/50-storage.md:383-388` is about counter *overflow* checks, not sweeper coverage.
- The S analog the finding cites is real: `crates/bumbledb/src/verify_store/counters.rs:49-58` convicts `stored <= tally.max_row_id` as `RowIdHighWaterLow`, and 63-82 convicts an *absent* S entry for any tallied relation. `crates/bumbledb/src/storage/read/row_id_high_water.rs:10-12` advertises this contract in prose ("the sweeper convicts a store where a live row id reaches it"). No such sentence can be written for Q today.
- The Q invariant is the spec, not an inference: the key-layout table in `docs/architecture/50-storage.md` (Q row, lines 56-61) states the ratchet law — "an EXPLICIT fresh-field value advances the high-water past itself ... so a copied id can never collide with a later mint; a correctness law of fresh semantics." Maintained at `crates/bumbledb/src/storage/delta/insert.rs:84-103` (`advance_fresh_marks`: `mark.next = mark.next.max(value.saturating_add(1))`), persisted at `crates/bumbledb/src/storage/commit/write.rs:355-358` (`flush_counters`) and 302-322 (`flush_escaped_fresh_ids`, the counters-only abort flush). `crates/bumbledb/src/storage/delta/alloc.rs:26-37,73-81` reads the stored Q value and hands it to the host verbatim (missing entry reads as 0), so a too-low Q directly mints a duplicate.
- The test suite pins the asymmetry exactly as claimed: `crates/bumbledb/src/verify_store/tests.rs:428-434` (`malformed_keys_in_every_swept_namespace_are_contextual_findings`) plants bare-tag keys for `NS_FACT, NS_MEMBERSHIP, NS_DETERMINANT, NS_REVERSE, NS_STAT` — Q is absent from the roster because planting it would produce no finding.
- The F pass already decodes every fact against the schema and the schema exposes `Generation::Fresh` per field (insert.rs:91-96 iterates `relation.fields()[i].generation` with the layout offsets in hand), so the tally is genuinely free machinery on the existing scan.

### Failure scenario

A store whose Q entry for `(users, id)` reads 5 while committed facts carry fresh ids up to 99 — torn restore, external tamper, the exact corruption class every other sweeper pass targets — returns an **empty** findings list from `verify_store`, contradicting its own contract ("a desynced store returns `Ok` with a populated report", verify_store.rs:52-55, and the doc's framing of the sweeper as the compensating control, 50-storage.md:375-381). The next `alloc()` reads Q=5 and hands the host id 5; if the original id-5 fact was deleted, or the id lives only host-side as an external identifier (the serial/newtype pattern from the mintable-pins doctrine), the re-issue is fully silent — the Lean-proved never-reissue law violated with no detection path online or offline.

### Suggested fix

During the F pass, tally `max_fresh: BTreeMap<(RelationId, FieldId), u64>` for `Generation::Fresh` fields (the fact bytes and layout are already decoded there). Add a Q pass mirroring `counters.rs`: cursor `[NS_FRESH, NS_FRESH+1)`, convicting malformed key length, unknown relation/field, and closed-relation entries like every other namespace; then convict a stored next-value that fails the ratchet law as a new `FreshNextValueLow { relation, field, stored, max_fresh }`, and convict an absent Q entry for a tallied fresh field exactly as absent S counters are convicted today (stored reads as 0). One correctness caveat the finder's proposed `stored <= max_fresh` misses: an explicit `u64::MAX` fresh value is legal (insert.rs:98-100 — `saturating_add` leaves `next == value == u64::MAX`, the sequence exhausted), so the conviction must be `max_fresh != u64::MAX && stored <= max_fresh` (equivalently `stored < max_fresh.saturating_add(1)` with the MAX case exempted) or a legal exhausted sequence is falsely convicted. Extend `malformed_keys_in_every_swept_namespace_are_contextual_findings` with a bare `NS_FRESH` key and add a desync fixture in the style of the existing counter fixtures.
