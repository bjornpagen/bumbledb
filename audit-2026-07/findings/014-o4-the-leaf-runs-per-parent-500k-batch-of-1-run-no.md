## o4: the leaf runs per parent — 500k batch-of-1 run_node calls for a fanout-1 determinant lookup

category: perf | severity: high | verdict: CONFIRMED | finder: perf:olap-temporal
outcome: deferred: no commit landed the batched leaf; the leaf still runs per parent — o4 moved only 0.95x vs the night in the wall-power rerun (campaign-2026-07-23/scenarios/scenarios.md, the DELTA table)

### Summary

Free Join execution batches every middle node across parents (the pump/probe_pass pipeline), but the last plan node always runs per parent: for each surviving parent the executor restores a full bindings row, reassembles cursors, and invokes `run_node` on a batch of exactly one. For `o4_segment_category` the leaf is Customer's `[segment]` trie level under a customer id already probed at the previous node — a key-covered lookup with fanout exactly 1 — so the batch machinery degenerates into 500,000 single-element leaf invocations plus 500,000 per-row group probes. The lane runs at ~59ns/row (night p50 29.7ms) where o1's kernelized scan-fold regime does ~0.9ns/row over the same 500k-row fold volume. Under the repo's representation-first doctrine this is a special case (the fanout-1 determinant lookup) handled by per-row control flow that a better representation — a batched leaf pass, or lowering the key-determined leaf to a value gather in the parent's pass — would erase.

### Evidence (all verified against the code; introspection reproduced)

- **The per-parent leaf arm**: `crates/bumbledb/src/exec/run/probe_pass.rs:529-551` — inside the per-survivor loop, `if leaf { bindings.load_row(...); for subatom { self.cursors[occ] = (assemble(occ), ...) } ... self.run_node(plan, node_idx + 1, ...) }`. The comment states it outright: "The last node runs per parent through the ordinary machinery." `crates/bumbledb/src/exec/run/pump.rs:30` carries the matching assert: `debug_assert!(node_idx + 1 < n_nodes, "the leaf runs per parent")`.
- **Reproduced plan shape**: I rebuilt the o4 query against the Olap schema (`crates/bumbledb-bench/src/scenarios/olap.rs:253-284`) on a 1/250-scale store and ran `Snapshot::introspect`. Output: node 2 = `subatom 0: occ 1 vars [0] cover(true)` with `estimated=2000 actual=2000 entries=2000` — the leaf is Customer's segment level, one node entry per Sale row, fanout 1 (node 1 shows `probes hit=2000` on occ 1's id level: the Customer cursor is already positioned one gather away). At bench scale (SALES = 500_000, olap.rs:96) this is the finder's `entries=500000` verbatim. `distinct_bindings: proven` also appears — the dedup seen-set is elided, so the per-row cost is purely leaf ceremony + group fold.
- **The batch-of-1 pinned arm**: `crates/bumbledb/src/exec/run/leaf.rs:49-107` — `run_leaf_pinned` is documented as "a batch of exactly one" and builds `LeafBatch { survivors: &[0], .. }` (leaf.rs:92), then calls `sink.emit_batch` per parent.
- **Per-call sink overhead**: `crates/bumbledb/src/exec/sink/aggregate/sink.rs:196` — `emit_batch` calls `self.refresh_shape_cache(batch)` unconditionally on every call (per-slot work, per row at batch size 1). One citation correction: the finding cited sink.rs:200 and read fold_batch.rs:23-24's "memoized across consecutive batches" as a claim about the shape cache — that comment describes the group-probe memo, which fold_batch.rs:73-75 records as deleted ("the probe IS the fast path now"). The substance is unchanged: nothing is memoized across these calls, and every "per batch" cost is per row.
- **Per-row group probe**: o4's group key is (segment, category) and segment is the leaf's key var, so `cached_constant_group` is false (`groups.rs:39-41`) and `emit_batch` takes `fold_batch_rows` (`sink.rs:212`) → `fold_scratch_row` → `load_group_key` + `probe_group`, a 2-word WordMap probe per row (`fold_row.rs:47-50`, `groups.rs:49-55`).
- **The numbers**: `bench-out/night-2026-07-20/scenarios/scenarios.md:35` — o4 at 29,701.8us over 500k fact rows ≈ 59ns/row; o1 at 445.9us ≈ 0.9ns/row (scenarios.md:31).
- **The determinant proof exists**: `crates/bumbledb/src/plan/fj/provably_distinct.rs:50-75` computes, per occurrence, whether the bound fields cover a key — for o4's Customer occurrence, bound fields {id, segment} ⊇ key {id}, which is exactly the knowledge that the leaf has fanout 1 (the o4 introspection's "distinct_bindings: proven" runs through it).
- **Paper check** (`docs/free-join-paper/arXiv-2301.10841v2/tex/04-optimizations.tex` § Vectorized Execution): the paper's vectorized algorithm also ends each batched probe pass with a per-tuple recursive `join` call — bumbledb's middle-node pump already improves on the paper, and this finding is the remaining residue of the paper's per-tuple recursion, not a divergence from the spec. It is an improvement opportunity beyond the paper, consistent with the paper's own locality argument for batching.

### Bench impact

o4_segment_category spends the bulk of its 29.7ms on 500k per-parent leaf invocations (slot-row copy at probe_pass.rs:533, cursor assembly at 539-550, dispatch, `gather_row`, `emit_batch` shape-cache refresh) plus 500k per-row 2-word group probes. Batching the leaf across parents — or absorbing the key-determined dimension lookup as a gather in the parent node's own batched pass — removes the per-row call ceremony and lets the group fold see real batches; the same per-parent tax prices j4/j5 and every dimension-lookup (star-schema FK) edge in the join and OLAP lanes. The contrast lane (o1 at 0.9ns/row through the scan-fold kernels) bounds what the machine does when the representation lets kernels run.

### Suggested fix

Two composable representation-level changes:

1. **Pipeline the leaf like every other node**: accumulate leaf-bound parents in the leaf's scratch (the machinery `probe_pass` already uses for middle nodes) and run one batched leaf pass. The sink handoff is the only new surface: `LeafBatch` currently assumes one constant outer binding row per call, so a cross-parent batch needs either per-parent outer rows in the batch or a sink entry point keyed on (parent rows × leaf keys).
2. **Lower key-covered determined leaves to a gather**: when an occurrence's bound fields cover a key (the `provably_distinct.rs:50-75` computation, already run per occurrence) and its only unbound vars are the key-determined remainder, the occurrence is not a join node — it is a value gather with fanout ≤ 1. Emit it as a gather step inside the parent node's batched pass (the sibling probe at the parent already positions the cursor; the leaf's entire job is one `gather_row`), and let the plan's last real node become the fold site. This is the star-schema FK-lookup shape; o4, j4, j5, and the rest of the o-family all take it.
