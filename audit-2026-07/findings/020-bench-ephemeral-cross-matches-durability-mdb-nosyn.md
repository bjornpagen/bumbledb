## bench --ephemeral cross-matches durability: MDB_NOSYNC engine vs fullfsync SQLite in the write families

category: bench-honesty | severity: high | verdict: CONFIRMED | finder: bench:honesty
outcome: fixed e5f35cb2 + 60d11a4d

### Summary

The `bench` subcommand's write/cold families (`commit_single`, `commit_batch`, `cold_containment_walk`, `cold_containment_walk_delete`, `bulk`) pair the engine against a SQLite oracle whose durability pragmas are hard-pinned at the *durable* level, regardless of `--ephemeral`. Under `bench --ephemeral` the engine scratch store is built with `Db::ephemeral` (`MDB_NOSYNC`: pages pwritten, no sync boundary ever crossed) while the oracle stays at WAL `synchronous=FULL` + `fullfsync=ON` + `checkpoint_fullfsync=ON` (F_FULLFSYNC on macOS on every commit). This is exactly the cross-matched pair the codebase's own `duralane.rs` declares "unrepresentable ... by type" — and it directly contradicts the lane-registry spec, which promises the ephemeral bench lane runs SQLite "minus the sync boundary." The `merge` subcommand then min-merges write rows across arbitrary run directories with no durability check, so mixing durable and ephemeral run dirs silently publishes the no-sync engine p50 as the per-family minimum.

### Evidence (all verified in-repo)

- `crates/bumbledb-bench/src/driver/write_families.rs:25-30` — the seam takes a bare `StoreMode`, not a `DurabilityLane`.
- `write_families.rs:72-75` — `mode.create(&scratch.join("db"), Ledger)` (engine side follows the flag) beside `corpus::load_sqlite(&scratch.join("oracle.sqlite"), cfg)` (oracle side does not). No lane pragma application follows; the same `conn` feeds all four PAIRED runners (lines 76-89). `bulk` (lines 119-131) pairs `bulk_bumbledb(cfg, scratch, mode)` with `sqlite_run::bulk(cfg, scratch)`, and `sqlite_run/bulk.rs:29` re-applies `corpus::configure_sqlite` (FULL) to every throwaway file.
- `crates/bumbledb-bench/src/corpus.rs:83-89` — `configure_sqlite` unconditionally sets `synchronous=FULL`, `fullfsync=ON`, `checkpoint_fullfsync=ON`.
- `crates/bumbledb-bench/src/cli.rs:56-58` and `driver/bench.rs:150-154, 268` — `--ephemeral` → `StoreMode::Ephemeral` → passed to `write_families`. `crates/bumbledb/src/storage/env/open_env.rs:68` confirms the ephemeral environment carries `EnvFlags::NO_SYNC`.
- `crates/bumbledb-bench/src/duralane.rs:1-21` — "a cross-matched pair ... is unrepresentable — the lane value carries both sides' config"; its `Nosync` arm (lines 100-110) applies `synchronous=OFF fullfsync=OFF checkpoint_fullfsync=OFF`, and `assert_parity` (line 127) is the readback. Grepping every `synchronous` call site in the bench crate: the OFF arms live only in `duralane.rs` (used by `crud`), `lanes/writes.rs` (used by the `writes` subcommand), and the churn lane — none reachable from the `bench` subcommand.
- **Spec divergence** — `docs/architecture/61-bench-lanes.md:70` defines the `bench-ephemeral-r1..r3` lane's SQLite parity config as: "the same session minus the sync boundary: `synchronous=OFF`, `fullfsync=OFF` — the honest `MDB_NOSYNC` twin." The code never implements this; the doc is the spec and the code diverges from it.
- Irony pin: `sqlite_run/fairness_check.rs:48-66` asserts `synchronous=2` and `fullfsync=1` unconditionally — its own comment says a write comparison at mismatched synchronous levels "is a lie" — and it runs on ephemeral runs too (`driver/bench.rs:187`), *enforcing* the cross-match rather than the pairing.
- Merge amplification: `crates/bumbledb-bench/src/report/merge.rs:52-53` extends the row pool with `merge_rows(parsed, "writes")` per run and takes min-of-runs p50/p95; `driver/merge.rs:11-27` accepts arbitrary run dirs and never reads `config.store`. The min column carries no durability label.
- Current-publication caveat (verified): `scripts/bench_viz.py` `derive_pools` (~lines 472-487) classifies runs by `config.store` and charts the durable pool when non-empty, so the README's published write numbers are not currently wrong; the write ladder the README quotes comes from `lanes/writes.rs`, which is correctly lane-paired. The dishonest rows nonetheless land in `report.json`/`report.md` on every `--ephemeral` run, and `README.md:337` documents `bench --ephemeral` ×3 as a standard protocol step.

### Failure scenario / Bench impact

Every `bumbledb-bench bench --ephemeral` run (three per documented night) emits write-family rows where ours pays zero sync and theirs pays F_FULLFSYNC — `commit_single` reads as a fake orders-of-magnitude win in the `| family | ours p50 | sqlite p50 |` table, with only a run-level "ephemeral stores" config line as a hint and no per-row durability label. `bumbledb-bench merge r1 r2 eph1` then mins the no-sync engine p50 into the per-family minimum column with no marker at all. Anyone quoting a merge table or an ephemeral `report.md` write row publishes a cross-matched number the suite's own doctrine calls a lie.

### Suggested fix

Thread `duralane::DurabilityLane` through the bench driver's write-family seam instead of `StoreMode` (`StoreMode` is derivable via `lane.store_mode()`), apply `lane.configure(&conn)` to the scratch oracle after `corpus::load_sqlite` (and inside `sqlite_run::bulk`'s per-file setup), and run `lane.assert_parity(&conn)` before timing — the closed-sum representation and its readback already exist; this call site kept the un-paired half of the axis. Additionally, either make `merge` refuse (or partition) run dirs whose `config.store` labels differ, or carry the store label onto each merged write row. This is a representation-over-control-flow fix in the project's own terms: the illegal state (a cross-matched twin) is already unrepresentable in `DurabilityLane`; the bug is a seam typed too weakly to use it.
