## pump's tail drain fires on every mid-stream recursion, collapsing batch means at depth ≥ 2 in ≥ 4-node plans — but no bench lane ever builds one

category: perf | severity: low | verdict: PLAUSIBLE | finder: engine:run
outcome: fixed 0f13feff + 2f326193 (R22: the >=4-atom bench shape landed)

### Summary

`pump` ends by draining its child's partial pending buffer (`crates/bumbledb/src/exec/run/pump.rs:198-200`). That drain is the right move exactly once — on the final unwind of the single `pump(0)` call (`execute.rs:449`) — but it also fires on every mid-stream `pump` invocation minted by `probe_pass`'s flush check (`probe_pass.rs:589-591`). In a plan of ≥ 4 nodes, every mid-stream `pump(1)` force-drains `scratch[2]`'s sub-batch remainder on return, so node 2 never accumulates survivors across `pump(1)` invocations; the effect cascades to every deeper middle node. This contradicts the module headline (`run.rs:20`: "The paper's 'cross-node-entry accumulation is future work' caveat is retired: deep nodes see full batches") for depth ≥ 2. Two corrections to the original claim, both verified: (1) **no benchmark exercises this regime** — every gated family including `chain` is ≤ 3 atoms, and node count equals occurrence count, so the premature drain fires on zero bench lanes; (2) the module's own measured pricing caps the batch-mean lever class at ~0.2–1.2%.

### Evidence (verified)

- `crates/bumbledb/src/exec/run/pump.rs:196-200` — the unconditional tail drain: `if node_idx + 2 < n_nodes && self.scratch[node_idx + 1].pending_len > 0 { self.pump(tables, plan, node_idx + 1, ...) }`. It runs on every `pump` return.
- `crates/bumbledb/src/exec/run/probe_pass.rs:589-591` — the mid-stream flush: `if !leaf && self.scratch[node_idx + 1].pending_len >= self.batch { self.pump(...) }`. Each such `pump(k)` (k ≥ 1) hits the tail drain when `k + 2 < n_nodes`, i.e. from 4-node plans up.
- `crates/bumbledb/src/exec/run/execute.rs:420-450` — `run_pipeline` seeds one virtual-root entry and calls `pump(0)` exactly once; `pump` only recurses downward, so the tail drain at node 0 is the one legitimate final drain.
- `crates/bumbledb/src/exec/run.rs:493-496` — pending buffers are documented "live across passes, reset per execution"; entries are self-contained (bindings row, carried `Cursor`s — stable pool indices per `colt.rs:63-66,100-101` — and origins), and cancellation is checked at processing time (`pump.rs:64-67`), so deferred accumulation across pump invocations is semantically safe under the module's declared set semantics (`run.rs:16-19`: cancellation is "correctness-free").
- `crates/bumbledb/src/exec/run.rs:20` — the headline claim the drain falsifies at depth ≥ 2 in ≥ 4-node plans.
- Paper check (`docs/free-join-paper/arXiv-2301.10841v2/tex/04-optimizations.tex`, § Vectorized Execution): the paper's algorithm recurses per surviving tuple after each batch — batching only within one node's probes. bumbledb's pipeline retires that caveat by accumulating pending rows per node; the mid-stream tail drain partially un-retires it at depth ≥ 2.
- **Correction 1 — no bench lane reaches the regime**: `binary2fj` (`crates/bumbledb/src/plan/fj/binary2fj.rs:18-65`) emits exactly one node per participating occurrence; `factor` (`plan/fj/factor.rs:11-33`) only hoists lookup subatoms between existing nodes, never changing node count. The gated `chain` family is exactly 3 atoms (`crates/bumbledb-bench/src/families/read.rs:92-131`: Posting–JournalEntry–Account), triangle is 3, every other read family is ≤ 3 (verified by per-function atom counts), rings/closure rules are ≤ 3, and querygen's `chain` shape is 2–3 hops (`querygen/shapes.rs:126-146`; only the non-perf differential `satellites` shape reaches 4 atoms). There is no "chain lane at ≥ 4 atoms".
- **Correction 2 — the repo already priced this lever**: `pump.rs:36-48` records that per-pass overhead measured 11–30 ns ("TWENTY TIMES below the assumption"), making "the whole batch-mean lever class a ~1% effect"; "lifting batch means to ~128 is worth 0.2–1.2% of triangle p50 … the lever class is closed." That measurement was on 3-node plans (where the premature drain never fires), so it does not literally cover ≥ 4-node chains, but the priced per-pass overhead is the same cost the extra partial passes would pay.

### Bench impact

None on the current corpus: the premature drain is dead code on every gated lane (all plans ≤ 3 nodes; the drain needs ≥ 4). For real user queries of ≥ 4 atoms with per-hop fanout·selectivity < 1, node 2's probe-batch means fall from ~128 toward 128·(fanout·selectivity) and compound at node 3+ — a real effect whose cost, per the module's own 11–30 ns per-pass pricing, lands in the low-single-digit-percent class, not the headline regression the original finding implies. The prefetch floor (`run.rs:421`, `PREFETCH_WIDTH_FLOOR = 4`) keeps MLP alive down to 4-element passes, further bounding the loss.

### Suggested fix

The drain is a property of the execution's tail, not of every pump return — reify that in the one caller (the representation-first move; `docs/design/representation-first.md`): delete `pump.rs:198-200` and, after `self.pump(&tables, plan, 0, ...)` in `run_pipeline` (`execute.rs:449`), drain in increasing node order: `for i in 1..n_nodes - 1 { if self.scratch[i].pending_len > 0 { self.pump(&tables, plan, i, ...) } }` (increasing order is required: draining node i appends node i+1's remainder). The bounded-memory contract is untouched — `probe_pass`'s own flush check (`probe_pass.rs:584-591`) still caps every pending buffer at two batches transiently. Also true up `run.rs:20`, which currently overclaims. Given the closed lever class and the empty bench regime, this is a doc-honesty + user-query-shape cleanup, not a bench win; if pursued, add a ≥ 4-atom chain shape to a lane first so the change is measurable.
