## Measure-filter path re-partitions and deep-clones the filter list per view build, leaking the pooled survivor buffer

category: perf | severity: medium | verdict: CONFIRMED | finder: cross:branching
outcome: fixed e76f0926

### Summary

`image::view::apply` re-derives at execution time a fact the plan already owns: whether and which of an atom's predicates are measure kinds (`DurationCompare`/`DurationFieldsCompare`). On every view build for a measure-carrying atom it (1) allocates a fresh `Vec` and deep-clones every non-measure predicate, and (2) when the atom's filters are *all* measures, silently drops the caller's pooled survivor buffer and reallocates it from zero. Both are unconditional heap hits on a path the engine's own allocation contract (docs/architecture/40-execution.md § The allocation contract) declares allocation-free: "a warm prepared-query execution performs zero heap allocations unless its intermediate sizes exceed every prior execution's", and the gate protocol adds "every repeat of a previously-seen parameter ... is allocation-silent". The clone allocates regardless of any high-water. This is also an inappropriate-branching finding under the repo's representation-first doctrine: three duplicated runtime `is_measure` scans re-compute a property that list *order* at lowering would encode for free.

### Evidence

All verified against the working tree:

- `crates/bumbledb/src/image/view/apply.rs:325-330` — `let is_measure = |p| matches!(p, DurationCompare{..} | DurationFieldsCompare{..})`; line 331 scans `predicates.iter().any(is_measure)` per call.
- `crates/bumbledb/src/image/view/apply.rs:340-344` — `let others: Vec<FilterPredicate> = predicates.iter().filter(|p| !is_measure(p)).cloned().collect();` — one Vec allocation plus a deep clone of each non-measure predicate per view build. `FilterPredicate` payloads are heap-owning: `Const::Words(Box<[u64]>)`, `Const::WordSet(Vec<u64>)`, `Const::PendingIntern{bytes: Box<[u8]>}` (`crates/bumbledb/src/image/view.rs:35-77`). The bind layer explicitly forbids exactly this ("a `Box` clone is a heap hit the steady-state contract forbids", `crates/bumbledb/src/api/prepared/bind.rs:538-541`) and pools its `Words` writes to avoid it.
- `crates/bumbledb/src/image/view/apply.rs:469-471` — `apply_infallible`'s `if predicates.is_empty() { return View::All(Arc::clone(image)); }` drops the moved-in `buf`, whose own doc four lines up (apply.rs:299-300) promises "caller-owned, reused across executions — capacity is retained". `View::recycle` returns `Vec::new()` for `All` (`crates/bumbledb/src/image/view.rs:290-295`), so the capacity never re-enters the `spare_buffers` ping-pong.
- `crates/bumbledb/src/image/view/apply.rs:384-386` — `refine_measure`'s `View::All` arm then allocates `let mut positions = Vec::new()` and grows it through the kernel. Net effect per measure-only rebuild: one pooled buffer freed, one fresh buffer allocated and grown.
- Call path: `crates/bumbledb/src/api/prepared/run_join.rs:115` (Idb, every fixpoint round) and `:155` (EDB, every memo miss) call `apply` with pre-resolved filters and `params = &[]` — param values are inlined into the filter list, so `ViewMemo::bind`'s list-equality check (`crates/bumbledb/src/api/prepared/view_memo.rs:27`) misses whenever the bound params leave the memoized set. One correction to the finder: `MEMO_SLOTS = 4` (`crates/bumbledb/src/api/prepared.rs:463`, active + 3 parked under LRU) absorbs a rotation of up to four distinct bindings per occurrence, so the leak fires on rotations *beyond* four bindings or on storage-generation advance — not on literally any churn.
- `crates/bumbledb/src/plan/fj/split_filters.rs:22-31` — a measured atom's *whole* filter list is pinned residual (its Eq constants stay view filters instead of becoming trie selections, per the filter-order law, docs/architecture/20-query-ir.md § the measure). This both maximizes the cloned set and guarantees param-carrying measured atoms rebuild through this exact path.
- `crates/bumbledb/src/exec/dispatch/classify.rs:45-50` — the third duplicated `is_measure` matches-scan (key-probe fast-path refusal), same plan-time fact recomputed.
- `crates/bumbledb/tests/alloc_gate.rs` — the CI allocation gate contains no Duration/measure scenario (grep for `Duration`/`measure` finds only the word "measured" in window comments), so nothing falsifies the contract on this path.

The in-code comment at apply.rs:335 calls the measure path "cold by shape", but the allocation contract grants no such exemption, and split_filters' residual pinning makes the path recur per execution for any measured atom whose params rotate past the memo.

### Failure scenario / Bench impact

A prepared query with `Duration(v) < ?p` plus other filters on the same atom (all of which split_filters keeps residual, Eq constants included), executed warm while `?p` rotates through more than four distinct values — or any measured atom inside a fixpoint (run_join.rs:115 rebuilds every round): every such execution allocates one `Vec<FilterPredicate>` plus a deep clone of each non-measure predicate, including any `WordSet`/`Words`/`PendingIntern` heap payloads. If the atom's residual list is measure-only, each rebuild additionally frees the pooled survivor buffer and re-grows a fresh one from zero. Under the documented CI gate protocol this is a straight contract violation: repeats of previously-seen parameters allocate. The gate simply has no measure scenario to catch it.

### Suggested fix

Reify the plan-time fact as list order: normalization/lowering already owns the filter-order law, so emit each occurrence's filter list with measure kinds last and carry the split index in the plan (or as a `(infallible, measures)` pair — parse, don't re-validate). `apply` then runs `&filters[..split]` through `apply_infallible` and refines with `&filters[split..]` — no `any()` scan, no `filter().cloned().collect()`, no per-call partition. Thread `buf` through the `View::All` arm of `refine_measure` (pass it into `filter_duration_range_u64` as the output buffer) so the empty-infallible case keeps the pooled capacity instead of dropping it. The three duplicated `is_measure` closures (apply.rs:325, split_filters.rs:23-28, classify.rs:45-50) collapse into the one plan-time fact. Add a measure-filter scenario (param-rotated past MEMO_SLOTS) to tests/alloc_gate.rs so the contract covers the path.
