## encode_literal is the second per-field encoder the module's own doc says cannot exist

category: unification | severity: medium | verdict: CONFIRMED | finder: engine:encoding
outcome: fixed 7406bd74

### Summary

`encode_fact`'s doc comment states the fact encoding is `append_key_field` concatenated "(no second per-field encoder can drift)" — but `encode_literal`, 70 lines up in the same file, is exactly that second per-field encoder: an independent match over `Value` with its own Bool/U64/I64/FixedBytes/Interval byte-writing arms and a `matches!`-derived `fixed_width` boolean flag branched inside two of them. Its output must be byte-identical to `append_key_field`'s at four load-bearing sites, and that parity is maintained purely by discipline: no test asserts it, and the arms most likely to drift (fixed-width intervals, fixed bytes) are the ones with zero direct or indirect coverage. Under the repo's representation-first doctrine (`docs/design/representation-first.md`: a case handled by a flag/branch that a better representation erases is the canonical defect), the `fixed_width` boolean is an inappropriate-branching micro-instance — `ValueRef` already reifies the distinction as `FixedIntervalU64`/`FixedIntervalI64` variants.

### Evidence (all verified in-repo)

- `crates/bumbledb/src/encoding/encode.rs:158-160` — the doc: "each field through [`append_key_field`], so the fact encoding IS the field encoding concatenated (no second per-field encoder can drift)".
- `crates/bumbledb/src/encoding/encode.rs:86-113` — `encode_literal`: `let fixed_width = matches!(desc, TypeDesc::Interval { width: Some(_), .. });` (line 87), then interval arms branch on it (lines 94, 101) to write the 8-byte start vs the 16-byte pair; `Value::FixedBytes` goes through `encode_fixed_bytes` (line 92) while `append_key_field` uses `FixedBytesValue::padded()` (encode.rs:141) — two independent implementations of the same word-padded encoding (currently byte-identical, verified against `FixedBytesValue` at encoding.rs:65-105).
- Parity is load-bearing at four consumer sites:
  1. `crates/bumbledb/src/schema/validate.rs:906-910` — `encoded_literal` seals sigma literals via `encode_literal`; `sealed_satisfies` (validate.rs:975-989) and the commit judgment (`storage/commit/judgment.rs:216`) compare those bytes with `==` against `field_bytes` slices of stored facts, which were encoded through `encode_fact` → `append_key_field` (encode.rs:167-172).
  2. `crates/bumbledb/src/schema/validate.rs:1575` — closed-relation sealed axiom rows are full facts built by concatenating `encode_literal` output, later byte-compared against dynamically committed facts.
  3. `crates/bumbledb/src/api/db/get.rs:93` — dynamic keyed-get probes encode key values via `encode_literal`; the adjacent comment asserts the parity as a contract: "exactly what `determinant_image` slices out of a stored fact" — and `determinant_image` slices facts encoded via `append_key_field`.
  4. `crates/bumbledb/src/schema/fingerprint.rs:270` — the schema fingerprint's canonical literal encoding.
- Test coverage: `encode_literal` appears in **zero** test files (grep over both crates). The only encoding parity test, `append_key_field_matches_determinant_image_slices` (`crates/bumbledb/src/encoding/tests.rs:214`), pins `append_key_field` against `determinant_image` — a different pair. Indirect coverage exists for the Bool/U64/str arms via commit integration tests (`storage/commit/tests/sealed_checks.rs:106,265` seal and match Bool and U64 literals end-to-end), but I found no test anywhere that seals a **fixed-width interval** or **fixed-bytes** selection literal and matches it against a stored fact: `tests/schema_macro.rs:514,532` and `schema/descriptor_codec.rs:573-583` assert descriptor shape only, and the fixed-width fixtures in `schema/tests/valid.rs:785` and `tests/exhume.rs:37` carry no selection literals. The branchy arms are exactly the untested ones.
- Feasibility of the fix: `ValueRef` (`crates/bumbledb/src/encoding.rs:120-142`) already carries `FixedIntervalU64`/`FixedIntervalI64` and `FixedBytes(FixedBytesValue)` variants, and `FixedBytesValue::padded()` produces byte-identical output to `encode_fixed_bytes` (raw bytes zero-padded to `⌈N/8⌉ × 8`), so delegation is a drop-in.

### Failure scenario

Not a live bug today — the two encoders currently agree byte-for-byte on every arm. The exposure is drift: any future change to one encoder's interval or fixed-bytes arm (reordering interval halves, changing the fixed-interval stored word, altering the pad) silently breaks byte-equality at commit — sealed sigma literals stop matching stored facts, closed-statement containments refute valid commits, dynamic keyed gets miss existing facts, and schema fingerprints shift — with no test to catch it on the fixed-width-interval and fixed-bytes arms.

### Suggested fix

Make `encode_literal` a lowering plus delegation: match `(value, desc)` to build the `ValueRef` — `Value::IntervalU64` + `TypeDesc::Interval { width: Some(_), .. }` → `ValueRef::FixedIntervalU64`, `width: None` → `ValueRef::IntervalU64` (same for I64), `Value::FixedBytes(raw)` → `ValueRef::fixed_bytes(raw)`, scalars to their variants — then call `append_key_field`. The second byte-writing match dies, `encode_fixed_bytes` becomes unreferenced by this path, the `fixed_width` boolean dies into the variant it was re-deriving, and the parity the doc at encode.rs:158-160 claims becomes structural (true by construction) instead of disciplinary. Optionally add one parity test asserting `encode_literal(v, desc)` equals `append_key_field(lower(v, desc))` across all arms, though after the delegation it pins the lowering rather than two encoders.
