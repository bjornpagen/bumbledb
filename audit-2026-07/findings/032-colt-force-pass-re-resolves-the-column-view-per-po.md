## COLT force pass re-resolves the column view per (position, column) — the exact cost the read path hoisted away

category: perf | severity: medium | verdict: CONFIRMED | finder: engine:colt
outcome: fixed b5e130d6

### Summary

`Colt::force_ingest` — the interior of the trie-force pass, which touches every position of a node once per level — decodes each key word through `word_at`, re-executing the full column-resolution chain (`View::image()` enum match, `RelationImage::column()` Box-slice index + enum match + slice construction with bounds checks, then a bounds-checked element read) for every (position, column) pair, plus a per-position scratch `clear/push/mem::take/restore` dance and a runtime-length `hash_words` fold. The trie's *read* path explicitly removed this exact cost — `gather_segment` hoists each column's slice once per segment and runs `get_unchecked` interiors, and `iter.rs` documents the win as "~1 load per (position, column) instead of an enum match and two bounds checks each." The build side of the trie is the unoptimized twin of its read side, and the Free Join paper names trie building a major bottleneck — the cost COLT exists to cut.

### Evidence (all verified against the code)

- `crates/bumbledb/src/exec/colt/force.rs:91-97` — per position: `self.scratch.clear(); for col in &self.schema_columns[level] { let w = self.word_at(*col, position); self.scratch.push(w); } let key = std::mem::take(&mut self.scratch); let hash = hash_words(&key);` and line 111 restores the scratch. `force.rs:46-64` shows this runs for every position of the node (root view walk or chunk-chain walk).
- `crates/bumbledb/src/exec/colt/count.rs:76-81` — `word_at` = `match self.view.image().column(column) { ColumnView::Words(words) => words[position as usize], ... }`.
- `crates/bumbledb/src/image/view.rs:249-254` — `View::image()` is an enum match per call; `crates/bumbledb/src/image.rs:189-198` — `column()` indexes `self.columns[column]` (bounds check), matches the `Column` enum, and constructs a range slice (bounds check), before the element read's own bounds check.
- The hoisted contrast: `crates/bumbledb/src/exec/colt/gather.rs:163-214` (`gather_segment`) resolves `self.view.image().column(*col)` once per (segment, column) and reads elements with `get_unchecked`; `crates/bumbledb/src/exec/colt/iter.rs:92-96` documents it: "each key column resolves its slice once per segment … ~1 load per (position, column) instead of an enum match and two bounds checks each."
- Const-arity precedent absent on the build side: `crates/bumbledb/src/exec/colt/probe.rs:86-94` dispatches `probe_walk::<1..=4>` per probe, and the const-arity hash exists (`hash_core::<K>` at `crates/bumbledb/src/exec/swar.rs:36`, exported as `hash_key_core` at `crates/bumbledb/src/exec/colt.rs:355`), yet `force_ingest` hashes with the runtime-length `hash_words` inside the one loop where arity is fixed for the whole pass.
- Same shape in growth: `crates/bumbledb/src/exec/colt/grow.rs:24-30` — `grow_map` re-gathers each key word-by-word into the scratch and re-hashes with runtime `hash_words`.
- Recurrence: `crates/bumbledb/src/exec/colt/new.rs:65-78` — `reset` clears `nodes/chunks/maps/ctrl/buckets/dense` on every generation rebind, so every trie re-forces from scratch; `crates/bumbledb/src/exec/colt/select.rs:27-28` — the union subtrie is truncated per execution, so set-selection executions re-force beneath it every time.
- Spec check: Free Join paper, `docs/free-join-paper/arXiv-2301.10841v2/tex/01-intro.tex:154-171` — "trie-building has been observed to be a major [bottleneck] … each trie is more expensive to build than a hash map"; `04-optimizations.tex` § COLT frames lazy building as the fix. The force pass IS the residual build cost the paper's own analysis says dominates — leaving its interior per-element cost class unoptimized diverges from the doctrine the read path already implements (representation-first: the segment-staged layout erases the per-element re-resolution).

### Bench impact

Every force in every join pays (positions × arity) repetitions of: a `View` enum match, an Arc-field load, a Box-slice bounds check, a `Column` enum match, a range-slice construction, and an element bounds check — where a staged gather pays one column resolution per segment and one bare load per element. This recurs on cold first executions, on every generation rebind (reset clears all forced maps), and on every set-selection execution (union subtries re-force each run). The scratch clear/push loop (a runtime-length loop with a capacity check per push) and the runtime-length hash fold add per-position overhead on top. One refinement to the finding: the per-position probe is already arity-monomorphic (`probe_hashed`'s internal `match key.len()` is a perfectly-predicted branch), so hoisting the dispatch is the smallest part of the win — the column-view hoist and bounds-check elision are the substantive mechanism.

### Suggested fix

Two-phase the force interior, mirroring the read path's `gather_segment` idiom: per chunk (or per fixed-size run of root positions), gather all key words column-hoisted into a staging buffer, then run the insert loop over staged rows with an arity-monomorphic ingest — dispatch once per force via the `probe_walk::<A>` pattern, hashing with `hash_core::<K>`. This also dissolves the per-position scratch take/restore (staged rows are plain slices) and satisfies the borrow checker, since the gather phase borrows `self.view` immutably before the insert phase mutates the pools. `grow_map`'s per-word re-gather (grow.rs:24-30) admits the same staged treatment.
