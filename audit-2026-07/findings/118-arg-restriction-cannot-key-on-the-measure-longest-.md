## Arg-restriction cannot key on the measure: "longest interval per group" is unspellable though every ingredient is paid for

category: missing-free-feature | severity: low | verdict: CONFIRMED | finder: cross:free-features
outcome: fixed 113d5f1e + db1d8f74 (R5)

### Summary

The interval measure (`Duration(v)` = end − start, type u64) is admitted as a fold input (`Sum`/`Min`/`Max` via `FindTerm::AggregateMeasure`), as a projected find (`FindTerm::Measure`), and as one side of an order comparison (`Term::Measure`) — but the Arg-restriction key position (`AggOp::ArgMax { key: VarId }` / `ArgMin`) admits only a rule variable of U64/I64. The composition "per group, the bindings attaining the extreme DURATION, with a carried payload" (e.g. "the longest outage per service, with its window" — the schema of cookbook recipe 1) is unspellable on every surface, and the obvious in-engine workaround (an interior predicate computing the per-group max duration, joined back with a measure comparison) is separately refused by `AggregateInteriorPredicate`. Meanwhile the execution sink already mints per-binding derived measure words and already runs the Arg sweep keyed on a scratch-row slot word, so keying that sweep on the computed measure reuses representation the machinery already pays for.

### Evidence (all verified against the code)

- **IR**: `crates/bumbledb/src/ir.rs:205-214` — `ArgMax { key: VarId }` with "`key` must be orderable (U64/I64)"; `ir.rs:247-256` — `AggregateMeasure` doc pins "the only three ops the measure admits (... `CountDistinct` and the Arg ops are typed rejections)"; `ir.rs:144-156` — `Term::Measure` legal only on one side of `Lt`/`Le`/`Gt`/`Ge`, never bindable, never under `Eq`.
- **Validation**: `crates/bumbledb/src/ir/validate/finds.rs:216-233` — the Arg key is checked as a bound variable whose resolved type is `U64 | I64` (`NonOrderableArgKey` otherwise); no measure arm exists.
- **Macro**: `crates/bumbledb-query-macros/src/lib.rs` (`parse_agg`) — `ArgMax(v, key)`/`ArgMin(v, key)` take exactly two idents (`expect_ident(... "the Arg key variable")`); the `Duration(v)` argument arm is gated to `Sum`/`Min`/`Max` with the typed parse error "the measure folds under Sum/Min/Max only".
- **TS**: `ts/src/query/find.ts:99-105` — `argMax<V extends AnyVar, K extends AnyVar>(value, key)`; the `FindEntryOk`/`FoldOverOk` judgments (find.ts:119-139) admit `Duration` only under `sum|min|max`.
- **Workaround foreclosed (join-back)**: the measure cannot be bound to a variable (`Term::Measure` illegal in atom bindings and under `Eq` — `ir/validate/context.rs:750-816` enumerates the legal comparison shapes), and the two-rule spelling — interior predicate `(service, Max(Duration(w)))`, output rule joining it with `Ge(Measure(w), Var(maxdur))` — is refused at `crates/bumbledb/src/ir/validate/validate.rs:180-191` (`AggregateInteriorPredicate`: fold-headed predicates legal only AT the output). So no program spells this.
- **Doctrine (checked per the audit's spec-first rule)**: `docs/architecture/20-query-ir.md` § "The measure" pins the legal positions "exhaustively" and rejects "any fold but the three". Notably the doc records explicit triggers for the adjacent deferrals — cross-rule Arg restriction ("a real sighting…") and `Sum∘Duration∘Pack` ("a measured two-pass budget violation") — but records **no trigger and no decision** for a measure-keyed Arg; the key position over the measure is simply not discussed. The gap is a doctrine pin by omission, not a weighed rejection.
- **Mechanism already paid for**: `crates/bumbledb/src/exec/sink/aggregate/new.rs:32-44` — sink construction already mints derived scratch words for every measure find (`derived = slot_count + measures.len()`, folded per row over the extended scratch row); `crates/bumbledb/src/exec/sink.rs:117-123` — `SinkSpec::Arg { key_slot, .. }` keys the restriction sweep on a slot word read from the same scratch row. Pointing `key_slot` at a derived measure word is exactly the existing composition; the `MeasureOfRay` poisoning path (sink.rs:146-153, `measure()` returning `None` for rays) already lives on this route.
- **Feature-register connection**: `docs/feature-register.md` § "The host-fold register" already watches an argmax host-fold family (`dissolveUnjudged`/`preCourseUnjudged` — "latest-attempt argmax"), under trigger 1(b) ("the host-fold register outgrowing one module"). The measure-keyed argmax is not yet itself a recorded citation there.

### Failure scenario

"Per service, the longest outage window" over cookbook recipe 1's schema (one fact per outage window, interval-typed): `ArgMax(w, Duration(w))` is a macro parse error, a TS type error, and unrepresentable in the IR; the interior-predicate join-back is `AggregateInteriorPredicate`. The host must project `(service, w, Duration(w))` (or `(service, w)` plus a `Measure` find) and arg-fold in application code — re-deriving the tie semantics ("a tie yields every attaining row") and the ray refusal that the engine already defines and tests for the variable-keyed Arg ops.

### Suggested fix

Widen the Arg key position from `VarId` to the same `var | Duration(var)` sum the fold input already has — either a two-case key enum on `ArgMax`/`ArgMin` or parallel `ArgMaxMeasure`/`ArgMinMeasure` ops mirroring `FindTerm::AggregateMeasure`. Execution reuses the existing derived-scratch-word minting (`sink/aggregate/new.rs`) so `SinkSpec::Arg.key_slot` points at the derived word, with the existing `MeasureOfRay` poisoning; validation adds an interval-typed check in the finds roster (finds.rs:216-233) beside `NonOrderableArgKey`; the macro's `parse_agg` admits `Duration(v)` in the key position; TS `argMax`/`argMin` gain a `Duration` key arm with an `IntervalVarOk` judgment. Amend `20-query-ir.md` § "The measure"'s exhaustive position list in the same change (the doc owns the enumeration). Alternatively, if deferral is preferred, record the composition in the host-fold register with an explicit trigger, as the neighboring Arg/Pack deferrals already do — today it is the only Arg-family unspellable with neither a spelling nor a citation.
