## NEON Allen kernels hold their memory-safety length invariant at debug strength, unlike every sibling unsafe kernel's release asserts

category: incoherence | severity: low | verdict: CONFIRMED | finder: r2:concurrency-unsafe-ffi
outcome: fixed 26f44a97

### Summary

The hand-NEON Allen configuration kernels (`crates/bumbledb/src/exec/kernel/neon.rs` — the module the kernel layer itself names "the one sanctioned unsafe module", kernel.rs:70-77) read 8-word windows through raw pointers from four independently-passed `&[u64]` streams, but the precondition that all four streams have length `n = codes.len()` is enforced only by `debug_assert!` — both inside the kernel (neon.rs:141-143 and 199-200) and at the public dispatch (allen.rs:105-107, `debug_assert_eq!`). In a release build nothing checks it: `allen_code_batch` sizes `codes` from `a_starts.len()` (allen.rs:104, 108) and passes `a_ends`/`b_starts`/`b_ends` through unchecked into `vld1q_u64` window loads (neon.rs:160-161, 174-176). Every sibling unsafe kernel in the same module pays for exactly this class of invariant at release strength: `fold.rs:22`, `fold.rs:61`, `fold.rs:93` use real `assert!` for the strided extent guarding `get_unchecked`, and `compact.rs:31` release-asserts the mask length while keeping only the correctness-not-safety 0/1 contract debug-only (compact.rs:33-36) and additionally hardening the unsafe loop with `keep & 1` (compact.rs:47). The Allen NEON module is also the one unsafe module Miri never interprets, so the debug-strength invariant sits precisely where dynamic verification coverage is weakest.

One correction to the original claim: the `n >= 8` precondition (whose violation would wrap `n - 8` at neon.rs:170 and read wildly OOB) is release-guarded — the dispatch branches on `codes.len() >= CODE_LANES` with a real branch before entering the NEON path (allen.rs:216, 232), so only the four-stream **length equality** is actually unenforced in release.

### Evidence (all verified against the working tree)

- neon.rs:139-143 — `let n = codes.len(); debug_assert!(n >= 8, ...); debug_assert!(a_starts.len() == n && a_ends.len() == n && b_starts.len() == n && b_ends.len() == n);` followed by the SAFETY comment asserting "every window reads 8 words from within the four n-length streams" and raw `vld1q_u64(b_s.add(base + lane))` loads (neon.rs:160-161) plus the overlapped tail at `n - 8` (neon.rs:170-182). Same pattern in `allen_code_batch_const_neon` (neon.rs:198-200).
- allen.rs:97-110 — public entry: `let n = a_starts.len(); debug_assert_eq!(a_ends.len(), n, ...); debug_assert_eq!(b_starts.len(), n, ...); debug_assert_eq!(b_ends.len(), n, ...); codes.resize(n, 0); codes_into(...)`. No release-strength check on the three non-`a_starts` streams anywhere on the path.
- Contrast, same module, same class of unsafe boundary: fold.rs:22 / fold.rs:61 — `assert!(stride > 0 && (count == 0 || (count - 1) * stride + offset < values.len()));` guarding `get_unchecked`; fold.rs:93 the min/max variant; compact.rs:31 — `assert!(mask.len() >= n);` guarding the unchecked cursor-write.
- Current callers are safe by construction: run_node.rs:336-352 and probe_pass.rs:306-320 both carve the four streams out of one `4 * n` scratch buffer with `split_at_mut(n)` three times, so equality holds structurally today.
- Miri gap: scripts/miri.sh:77-78 skips `exec::kernel::tests::allen` on the native aarch64 pass ("Miri does not interpret NEON intrinsics", miri.sh:36-43), and the cross x86_64 pass exercises only the scalar reference twins — the `neon.rs` bodies are never interpreted. The fold/compact unsafe, by contrast, runs under Miri in the `exec::kernel::tests` filter.
- Fix constraint the original suggestion misses: scripts/check-asm.sh (lines 49-56, 68-71) greps the release disassembly of the three `*_neon` symbols and fails on any `cmp`/`csel`/`adds`/`ccmp`/`bl` — a release `assert!` inside neon.rs compiles to a `cmp` plus a panic call and would trip the flag-free gate. The release check must live at the ungated dispatch layer (allen.rs), consistent with the module doc's claim that "flag-free asm gates forbid the bounds-check `cmp`" inside the kernels (kernel.rs:18-19).

### Failure scenario

A future crate-internal caller of `allen_code_batch` (the `exec` module is `pub(crate)`, lib.rs:101, so exposure is internal-only) passing a b-side stream shorter than `a_starts` — e.g. a gathered scratch reused across residuals without re-sizing — reads past the b slice in 8-word NEON windows in release: silent OOB reads feeding wrong Allen codes, with no assert (debug-only), no Miri (module never interpreted), and no reference-twin divergence unless a property test happens to construct the unequal-length shape (the existing tests, kernel/tests.rs `allen_corpus`, always build four equal-length vectors). This is a hardening/incoherence finding, not a live bug: both shipped callers are structurally safe today.

### Suggested fix

Promote the three stream-length equalities to release `assert_eq!` at the dispatch in `allen.rs::allen_code_batch` (and the two in `codes_into_const`'s public path) — one predictable branch per batch, outside the gated `*_neon` symbols so `scripts/check-asm.sh` stays green. Do **not** promote the copies inside `neon.rs`: the flag-free asm gate forbids the `cmp`/`bl` a release assert emits there; the inner `debug_assert`s can stay as documentation or be deleted. Better still, per the representation-first doctrine (docs/design/representation-first.md): make the invariant unrepresentable-when-false by passing the four streams as one type — e.g. an `EndpointStreams<'a>` whose constructor performs the single length check (or which is minted directly from the `split_at_mut` carve the two callers already do), erasing the precondition instead of asserting it.
