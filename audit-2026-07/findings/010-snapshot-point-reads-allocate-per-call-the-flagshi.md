## Snapshot point reads allocate per call — the flagship keyed-get lane violates the allocation-free point-read contract

category: perf | severity: high | verdict: CONFIRMED | finder: perf:points

### Summary

The snapshot point-read surface — `Snapshot::get_dyn`, `Snapshot::get`, and `Snapshot::contains_dyn` — stages its determinant (and, for `contains_dyn`, its value refs and fact bytes) in fresh heap `Vec`s on every call. This is the exact surface the `p5_keyed_get` benchmark lane times and the TS SDK bridge routes point reads through. The API contract says point reads are allocation-free, the `WriteTx` twins implement a hand-rolled scratch discipline citing that contract, and the alloc gate asserts it — but only for the `WriteTx` surface. The snapshot surface was never gated and allocates on every call. The determinant `Vec` is additionally a pure staging copy: the very next frame copies it into a stack buffer.

### Evidence (all verified against the working tree)

- `crates/bumbledb/src/api/db/snapshot.rs:179` (`get_dyn`) and `:235` (`get`): `let mut determinant = Vec::new();` per call. `contains_dyn` allocates twice more: `:145` `let mut refs = Vec::with_capacity(values.len());` and `:151` `let mut fact = Vec::new();`.
- The contract: `docs/architecture/70-api.md:537` — point reads are "determinant gets (allocation-free, no images, no plans)".
- The twin honors it: `crates/bumbledb/src/api/db/get.rs:226-233` (`WriteTx::get`) — "the determinant must not allocate per call (point reads are allocation-free, `docs/architecture/70-api.md`) — so encode into the taken buffer" — with an explicit `std::mem::take(&mut self.scratch)` / restore dance. `WriteTx::get_dyn` and `contains_dyn` route through `with_scratch`/`self.scratch` the same way (get.rs:270, :307-310).
- The gate gap: `crates/bumbledb/tests/alloc_gate.rs:1108-1149` (`borrowed_struct_gate`) asserts `(0,0,0,0)` allocations for `tx.insert` + `tx.get` inside `db.write`. The snapshot sections of the gate exercise only `snap.execute`/`snap.execute_args` (lines 936-1072, 1355) — no `snap.get`, `snap.get_dyn`, or `snap.contains_dyn` anywhere in the file.
- The timed lane: `crates/bumbledb-bench/src/scenarios/run_query.rs:53-59` — `Engine::KeyedGet` answers via `db.read(|snap| snap.get_dyn(*relation, *statement, params))`; `points.rs:318` names it `p5_keyed_get`. The TS bridge is the same surface: `ts/crate/src/lib.rs` (~line 617), `SnapReq::Get` → `snap.get_dyn`.
- The staging copy: `crates/bumbledb/src/storage/read/determinant_row.rs:22-24` — `let mut key: KeyBuf = [0; MAX_KEY]; let len = keys::determinant_key(&mut key, rel, statement, determinant);` — the heap Vec's bytes are immediately memcpy'd into a stack buffer after the `U|relation|statement` prefix.
- The stack buffer is total: `crates/bumbledb/src/storage/keys.rs:205` `MAX_DETERMINANT_WIDTH = MAX_KEY − R_OVERHEAD = 496`, enforced at schema construction (`crates/bumbledb/src/schema/validate.rs:572` rejects wider key projections at declaration), so no determinant can exceed a `[u8; MAX_KEY]` cursor.
- Alloc count per p5 hit, against the Doc relation (`points.rs:25-31` — `key: str`, `payload: bytes<32>`): determinant Vec (snapshot.rs:179) + the decode's `Vec<Value>` collect (`encoding/decode.rs:235-259`) + `Box<[u8]>` for the resolved string (decode.rs:247, via `Box::from(dict::resolve(...))` at snapshot.rs:202) + `Box<[u8]>` for the FixedBytes payload (decode.rs:248 `value.as_bytes().into()`) = 4 alloc/dealloc pairs. A hit at the U probe that misses (or any non-string-short-circuit miss) still pays the determinant alloc.

### Bench impact

`bench-out/night-2026-07-20/scenarios/scenarios.json`: `p5_keyed_get` ours p50 = 1416 ns vs SQLite 1417 ns, `ratio_p50` 0.9993 — the only lane in the points world (and the whole scenario pin) where SQLite ties us. Four allocator round-trips inside a 1.4 µs operation are a material slice of the gap; the determinant Vec + memcpy is pure representation waste (the bytes end up in a stack `KeyBuf` regardless), and this is precisely a representation-over-control-flow violation per `docs/design/representation-first.md`: the buffer that the composed key already needs is the right home for the determinant encoding — the intermediate heap staging exists only because the encoding and the key composition were written as two steps.

### Suggested fix

Fuse determinant encoding with U-key composition: give `encode_determinant_with` (api/db/get.rs:55) a writer target that is a stack `[u8; MAX_KEY]` cursor pre-positioned after the `U|relation|statement` prefix (a `KeyWriter` continuation of `keys::determinant_key`), then probe LMDB with the composed slice directly. `MAX_DETERMINANT_WIDTH = 496` being schema-enforced makes the stack buffer total — no spill arm needed. The closed-relation arm (`closed_fact_by_determinant`) still receives its determinant slice as `&key[prefix..len]`. `Snapshot::get` is `&self`, so no `&mut` scratch exists there — the stack cursor is the correct shape for both snapshot reads, and `WriteTx` can adopt the same representation and retire its take/restore dance. The `get_dyn` decode's three allocations are inherent to its owned `Vec<Value>` return; the typed `Snapshot::get` (whose fact borrows from the dictionary) becomes fully allocation-free with the determinant fix alone. Finally, extend the alloc gate: add `db.read(|snap| snap.get(key))` (and the dyn siblings) to `tests/alloc_gate.rs` so the snapshot surface is held to the same `(0,0,0,0)` contract as `borrowed_struct_gate` holds the `WriteTx` one.
