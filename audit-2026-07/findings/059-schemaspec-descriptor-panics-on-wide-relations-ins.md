## SchemaSpec::descriptor panics on wide relations instead of issuing a typed SpecIssue

category: bug | severity: medium | verdict: CONFIRMED | finder: theory
outcome: fixed ec859d81

### Summary

`SchemaSpec::descriptor()` — the spec-lowering entry point that is the runtime bindings contract for foreign hosts (the published Node bindings marshal plain JS data into a `SchemaSpec` and call it) — mints sealed `FieldId`s with `u16::try_from(sealed).expect("field count fits u16")` and has no field-count cap of its own. Any statement that names a field at sealed index >= 65536 panics inside lowering. The function's own `# Panics` doc justifies this by deferring oversized counts to the engine's typed `SchemaError::RelationTooManyColumns`, but that engine check runs only after `descriptor()` returns, so it is preempted in exactly the case that panics. On the Node path the panic unwinds into a napi-generated `extern "C"` trampoline with no `catch_unwind`, which under edition-2024 abort-on-unwind semantics is a process abort — instead of the promised one-round-trip complete `SchemaSpecError` issue list.

### Evidence (all verified against the working tree)

- `crates/bumbledb-theory/src/schema/spec.rs:529-532` — `Resolver::field`: `let sealed = index + usize::from(closed); Some(FieldId(u16::try_from(sealed).expect("field count fits u16")))`. Reached for any statement naming a field whose declared index (plus 1 for a closed relation's synthetic `id` shift) exceeds `u16::MAX`. No cap pass anywhere in spec.rs (grep for `u16::MAX`/`RelationTooManyColumns` finds only this expect and the doc mention).
- `crates/bumbledb-theory/src/schema/spec.rs:762-769` — the `# Panics` doc: "When a relation or field ordinal exceeds the id space (`u32`/`u16`) … the declaration boundary (the engine's `SchemaError::RelationTooManyColumns` / `SchemaError::TooManyStatements`) is where such counts are rejected typed." That boundary is `crates/bumbledb/src/schema/validate.rs:61-69`, inside `SchemaDescriptor::validate` — which runs only on the descriptor `descriptor()` never finishes building. (Nuance: if no statement addresses a far field, `descriptor()` succeeds — the relation-lowering loop at spec.rs:799-857 clones fields without minting ids — and the engine then rejects typed. The doc's model holds only for that sub-case; the statement-addressing case panics first.)
- **Reproduced empirically**: a temporary integration test (deleted after the run) built a `SchemaSpec` with one open relation of 65,537 `u64` fields (`f0`..`f65536`) and one `StatementSpec::Fd { relation: "wide", projection: ["f65536"] }`. `cargo test -p bumbledb-theory` under `catch_unwind` observed: `panicked at crates/bumbledb-theory/src/schema/spec.rs:531:35: field count fits u16: TryFromIntError(PosOverflow)`.
- Engine contrast — the check-before-mint ordering the spec path lacks: `crates/bumbledb/src/schema/validate.rs:42-50` ("Field and statement counts need no panic path: … checked before any u16 id is minted") and :56-69 ("The derived-column cap runs FIRST — before `materialized_statements` or the field loop mints any u16 id"), :72-75 ("never the id-mint expect").
- FFI exposure: `ts/crate/src/lib.rs:238` — `open_with` calls `spec.descriptor()` on the output of `marshal::schema_spec` (`ts/crate/src/marshal.rs:560-614`, which loops JS arrays into `FieldSpec`s with no count cap). The surrounding contract at lib.rs:274-276: "Schema resolution and validation failures return as data; environment failures throw." No `#[napi(catch_unwind)]` appears anywhere in `ts/crate/src`; in napi-derive-backend 5.1.2 `catch_unwind` is opt-in per attribute (`codegen/fn.rs:154-157`), so the plain `#[napi]` trampolines do not catch — and `ts/crate` is edition 2024 with no panic-profile override, so the unwind aborts at the `extern "C"` boundary.
- Doctrine the panic violates, per the in-repo spec material: `docs/architecture/70-api.md` (§ Foreign-host spec, ~line 291-296): `descriptor()` returns "the typed `SchemaSpecError`, which enumerates EVERY unresolvable name … in one pass (a foreign host repairs its whole spec in one round trip)"; same doc's ETL doctrine (~line 784): "ETL input is data, not code — no panics on the [wire-facing] path". The crate's own precedent for a shape lowering cannot represent is `SpecIssue::RowArityExcess` (spec.rs:280-294: "the lowering rejects here rather than truncate silently" — an issue, not a panic), and the placeholder-that-never-escapes pattern already exists at `Resolver::literal` (spec.rs:607-634, `Value::U64(0)` with "placeholders never escape (a nonempty issue list fails the whole construction)").

### Failure scenario

A foreign host (Node bindings, or any ETL tool emitting raw `SchemaSpec` data) submits a generated or adversarial spec containing a relation with 65,537 fields and any statement projecting or selecting the last field. `dbCreate`/`dbOpen` → `marshal::schema_spec` (succeeds) → `spec.descriptor()` → panic at spec.rs:531 → unwind hits the napi `extern "C"` trampoline → **process abort**. The host process dies instead of receiving `{ ok: false, kind: "schemaError", message: <complete issue list> }`; the engine's typed `RelationTooManyColumns` never fires because lowering never completes.

### Suggested fix

Mint a typed `SpecIssue` during lowering, following the two precedents the crate already owns:

- Preferred (mirrors validate.rs's check-before-mint ordering): an upfront cap pass over `spec.relations` at the top of `descriptor()` — for each relation, if `fields.len() + usize::from(extension.is_some()) > usize::from(u16::MAX)`, push a `SpecIssue::RelationTooManyColumns { relation, name, columns }`; the existing dead-expect at spec.rs:531 then becomes genuinely unreachable, matching its engine twin, and the `# Panics` section of the doc can be deleted for the field case.
- Alternative (the `literal()` precedent): at the failed `try_from` in `Resolver::field`, push the issue and return a placeholder `FieldId(0)` that never escapes (a nonempty issue list fails the whole construction), exactly as `Resolver::literal` does with `Value::U64(0)`.

Either way, update the spec.rs:762-769 doc — its current justification (typed rejection at the declaration boundary) is unreachable in the case that panics.
