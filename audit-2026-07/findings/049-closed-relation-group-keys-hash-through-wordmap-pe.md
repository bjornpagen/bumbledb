## Closed-relation group keys hash through WordMap per row; the dense ≤256 domain the schema already proves is unused

category: missing-free-feature | severity: medium | verdict: CONFIRMED | finder: perf:olap-temporal

### Summary

Every varying-group aggregate fold resolves its group via `probe_group` — a `WordMap` hash-and-probe per folded row. But for group keys whose spans are closed-relation- or bool-typed, the schema already proves a tiny dense domain: closed extensions are sealed at ≤256 rows (`MAX_EXTENSION_ROWS`), their member values are the declaration indices `0..N-1`, and containment enforcement guarantees every committed referencing value is in-domain. The group index for such keys is pure arithmetic (`seg*16 + cat`) into a flat accumulator table — no hash, no ctrl-line probe, no insert branch. The type information exists at prepare time in the very function stack that constructs the sink; it is never threaded into the grouping representation. This is the representation-first doctrine's exact shape (docs/design/representation-first.md: "change the data, types, and invariants so the case stops being special"): the map exists only because the domain is treated as open.

### Evidence

All citations verified directly; the plan-routing claim was verified empirically by running `Snapshot::introspect` on the real olap scenario at full 500k-row scale.

- **Per-row probe**: `crates/bumbledb/src/exec/sink/aggregate/groups.rs:49-55` — `probe_group` calls `self.groups.get_or_insert_with(&self.key_scratch, || next)` on every invocation. `crates/bumbledb/src/exec/sink/aggregate/fold_row.rs:47-50` — `fold_scratch_row` loads the group key and calls `probe_group` once per row. `crates/bumbledb/src/exec/sink/aggregate/sink.rs:204-213` — `emit_batch` routes any batch whose group span includes a leaf key slot (`cached_constant_group == false`) to `fold_batch_rows`, the per-row arm. The constant-group hoist (`fold_batch.rs:69-76`, one probe per batch) fires only when every group word is outer-bound.
- **Bench routing (measured, not assumed)**: introspection of the olap scenario at 500k Sale rows shows o3 (`promo_split`) plans as one node with trie schema `[[2,1,0]]` — the bool group var `v0` is a leaf key slot, so all 500,000 rows take the per-row probe; o4 (`segment_category`) plans 3 nodes with the leaf node covering exactly `v0` (segment, a group var) at 500,000 entries — per-row probes with 2-word keys. `distinct_bindings: proven` for both, so the binding seen-set is elided and the group probe is the *only* per-row hash left on these paths.
- **Dense domains are proven by the schema**: `crates/bumbledb-theory/src/schema.rs:319` — `MAX_EXTENSION_ROWS = 256`, enforced at `crates/bumbledb/src/schema/validate.rs:1513`; `validate_extension` (validate.rs:1538-1541) encodes each closed member's synthetic id as its declaration index, so members are literally `0..N-1`. `crates/bumbledb/src/schema.rs:296-338` — `AxiomIndex(u16)` + 4×u64 `MemberSet`. `crates/bumbledb-bench/src/scenarios/olap.rs:47-57` — `Segment` (4 members), `Category` (16), with `Customer(segment) <= Segment(id)` and `Product(category) <= Category(id)`; these compile to `Enforcement::Closed { members }` (schema.rs:280) checked at commit, so committed values are guaranteed in-domain — o4's group product is exactly 4×16 = 64 dense ordinals, and dense indexing cannot go out of bounds on committed data.
- **The information never reaches the sink**: `MemberSet` has zero uses under `src/exec/`. `AggregateSink::build` (`crates/bumbledb/src/exec/sink/aggregate/new.rs:147-258`) receives only `finds` and `slot_count` and unconditionally builds `groups: WordMap::with_capacity_hint(key_words, hint.min(4096))` (new.rs:221). Yet `make_sink`'s caller in `crates/bumbledb/src/api/prepared/build.rs` already holds `&Schema` and per-variable column types (build.rs:982 reads `column.ty` for the key-probe path) — the domain facts are in scope at construction and simply not threaded.
- **Docs check**: docs/architecture/40-execution.md's sink contract specifies the fold/dedup semantics and the dense-iteration rule but nowhere mandates an open-domain group map; the Free Join paper (docs/free-join-paper) governs the join, not the sink, so no spec conflict.

### Bench impact

o3's 500k one-word probes and o4's 500k two-word probes each become one multiply-add and one indexed load. Tempering: the `WordMap` is already a highly tuned SWAR map (one ctrl-line per probe step, 33% max load — `exec/wordmap.rs:1-30`), so the per-row saving is a few nanoseconds, not tens — a meaningful slice of o3's ~15ns/row all-in cost, not a multiple of it. The structural win generalizes: any GROUP BY over enum-like closed dimensions — the dominant OLAP grouping shape — gets the dense path for free, and the insert-time accumulator-seeding branch (groups.rs:57-85) leaves the hot loop entirely (dense slots can be pre-seeded at bind).

### Suggested fix

At sink construction (`api/prepared/build.rs::make_sink` and `AggregateSink::build`), when every group span's type resolves to a closed relation or bool, record the per-span radix (the sealed extension's row count; 2 for bool) and select a dense accumulator table of size Π radix (cap the product — 4096, the existing map-hint clamp, is a natural bound; overflow falls back to the map). `probe_group` becomes index arithmetic plus a touched-bitmap (or an `is_seeded` check) and `finalize` iterates touched slots reconstructing key words from ordinals. Open-typed spans keep the `WordMap`. The choice is bind-time data — an enum on the sink's group representation — never a hot-loop branch, matching the doctrine in docs/design/representation-first.md.
