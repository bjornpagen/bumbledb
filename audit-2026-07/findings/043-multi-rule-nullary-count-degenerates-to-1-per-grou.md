## Multi-rule nullary Count degenerates to Count = 1 per group; ir.rs contract and the docs' own worked example contradict the shipped union-regime semantics

category: incoherence | severity: medium | verdict: CONFIRMED | finder: lean:query
outcome: fixed 34f96cd8

### Summary

Under the multi-rule ("union") regime the engine dedups on the head projection, and the nullary `Count` head position contributes nothing to that key. For a head of `[group vars..., Count]` the union key is therefore exactly the group key: every binding of a group after the first is dropped before the fold, and every inhabited group answers `Count = 1` no matter how many bindings it has. The result is data-independent — a caller who writes a two-rule (or `Or`-lowered) counting query gets an existence test wearing a Count costume. The naive differential oracle encodes the same collapse (a constant filler in a set), so the pinned differential query passes while agreeing on the degenerate value. Meanwhile the normative contract in `ir.rs` states the fold domain with no across-rules carve-out, and the architecture doc's own worked example (`Sum ... = 200`) sits inside the across-rules clause under which it is false. The project already has a typed refusal (`ArgAcrossRules`) for exactly this hazard class — a fold whose needed information lives outside the head's vocabulary — but binding multiplicity, which Count and Sum consume, is not refused.

### Evidence (all verified against the working tree)

- **Engine mechanism.** `crates/bumbledb/src/exec/sink/aggregate/new.rs:379-393` — `union_span` returns `None` for `SinkSpec::Agg { over_slot: None, .. }` (nullary Count "reads nothing and contributes nothing", per its own comment at lines 373-377). `crates/bumbledb/src/exec/sink/aggregate/fold_row.rs:36-45` — under the union regime the seen-set keys `dedup_key(union_spans, ...)` (the head projection, fold_row.rs:167-181) and returns early on a repeat; fold_row.rs:80 — `(FoldOp::Count, Acc::Count(n)) => *n += 1` runs only for first-seen keys. With head `[account, Count]`, the key is the account words alone, so each group folds once.
- **Naive oracle agrees on the degenerate value.** `crates/bumbledb-bench/src/naive/query.rs:499-539` (`union_fold`): the fold domain is a `BTreeSet<Tuple>` of head-projected rows where `FindTerm::Aggregate { over: None, .. } => Ok(Value::Bool(false))` (line 534) — a constant filler, so a group's N bindings collapse to one tuple and `Count = 1`.
- **The differential pins this shape.** `crates/bumbledb-bench/src/verify/run_algebra.rs:111-134` constructs the two-arm `[Var(0), Count]` query ("The union folds: Count's constant-filler head position"). Engine and oracle agree on 1, so the harness green-lights the degenerate semantics.
- **The contradicted contract.** `crates/bumbledb/src/ir.rs:187-190`: "The fold domain of every aggregate is the group's set of distinct full bindings over all query variables" — no across-rules qualification; under the union regime the shipped fold domain is distinct head projections, a strictly coarser set.
- **The contradicted doc example.** `docs/architecture/20-query-ir.md:283-290`: the bullet's across-rules clause ("Across rules, aggregates read the head: the fold domain is the union of the rules' binding sets projected to the head ...") is immediately followed by "Two postings of amount 100 to one account are two distinct bindings (their fresh ids differ): `Sum(amount) by account` = 200." Under the head-projection fold both postings project to the single tuple (account, 100) — the answer is 100; the fresh ids cannot distinguish anything because they are not in the head. The example is true only in the single-rule full-binding regime described by the bullet's first sentence.
- **The Lean citation overreach.** The doc cites `lean/Bumbledb/Exec/Dedup.lean: union_regime_head_projection` as authority for the aggregate fold-domain sentence. The theorem (Dedup.lean:567-601) is stated over `(rule e).finds.map (bind e)` with `finds : List VarId` — projection answers only, no aggregate fold composition. The module doc itself records the gap (Dedup.lean:77-83): "the nullary `Count` head position contributes NO words to the union key ... a keyless head position is unrepresentable in the theorem's `VarId` finds; sound, since omitting a constant column never changes key equality" — sound for key equality, silent on the multiplicity the fold consumes.
- **The precedent refusal.** `crates/bumbledb/src/ir/validate/validate.rs:399-418`: Arg-restriction across lowered rules refuses with `ValidationError::ArgAcrossRules` precisely because "the restriction key is a rule-scoped variable outside the head's vocabulary"; binding multiplicity is equally outside the head's vocabulary, but Count/Sum pass validation.

### Failure scenario

`finds [account, Count]` with rules R1: `Posting(account=a, at=t), t < T1` and R2: the same atom with `t >= T2` (or one rule whose `Or` lowers to two). For an account with 1,000 postings across the two windows, both the engine and the naive model return `Count = 1`. The user's intent (postings in either window) and ir.rs's own fold-domain sentence both say ~1,000. The answer is data-independent for every multi-rule nullary-Count query; the differential harness cannot catch it because both oracles share the collapse, and run_algebra.rs deliberately pins the shape.

### Suggested fix

Refuse nullary `Count` (and value folds like `Sum` that consume binding multiplicity, per the companion finding) on programs that lower to 2+ rules, exactly as `ArgAcrossRules` does and at the same site (validate.rs:399-418) — the modeling answer is already established doctrine: one query per rule, host-merged. Independently, repair the two documents: give `ir.rs:187-190` the across-rules carve-out (or let the refusal make the sentence unconditionally true), and move or re-scope the "= 200" example in `docs/architecture/20-query-ir.md:283-290` so it no longer sits inside the across-rules clause it contradicts. If the refusal lands, also update the Dedup.lean module doc (lines 77-83) so the Count vocabulary-gap note records the refusal rather than framing the gap as benign.
