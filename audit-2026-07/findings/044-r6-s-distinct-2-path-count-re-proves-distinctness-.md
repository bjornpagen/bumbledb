## r6's distinct 2-path count re-proves distinctness per row in a seen-set that COLT forced maps already hold

category: perf | severity: medium | verdict: CONFIRMED | finder: perf:rings
outcome: deferred: the forced-map telescoped Count never landed; r6 honestly flat in the campaign rerun (scenarios/delta.md)

### Summary

`r6_two_path_count` (bare `Count` over the distinct (v0,v1,v2) 2-paths of `Transfer`) cannot obtain the plan's `DistinctWitness`, because the witness recognizes only fact-shape distinctness — bound fields covering a relation key — and `Transfer`'s only key is the fresh `id` the rule never binds. The aggregate sink therefore retains its binding seen-set and hash-inserts the full 3-slot binding of every one of ~5.3M emitted rows (~135ms at ~25ns/row, essentially the whole lane). But distinctness is exactly what the COLT trie computes when a level is forced: a forced map's keys are the distinct key set and its `len` is `KeyCount::Exact`. Duplicates exist in r6's emitted stream only because the paper's suffix rule iterates last-level nodes as raw row positions. Forcing both levels (amortized O(rows) once, node-shared) makes every enumeration path a distinct binding tuple — covers bind exactly the node's new vars — and the bare Count telescopes: per distinct (v0,v1), the leaf contribution is the forced dst-map's O(1) `len`. The lane drops from ~5.3M hashed inserts to O(edges + distinct pairs) ≈ 130K operations.

### Evidence (all verified in-repo)

- **The query never binds the key.** `crates/bumbledb-bench/src/scenarios/rings.rs:288-304` — r6 is two `Transfer` atoms binding `(src,dst)` and `(src,dst)` with `finds: count()`. `Transfer`'s schema (rings.rs:33-39) declares `id: u64 as RgTransferId, fresh` — the fresh id is the relation's one key.
- **The witness is fact-shape only.** `crates/bumbledb/src/plan/fj/provably_distinct.rs:40-75` — coverage requires each participating occurrence's bound/pinned fields to cover a key projection (`Functionality` statements). r6 binds `{src,dst}`, the key is `{id}` → witness `None`.
- **No witness ⇒ per-row seen-set inserts.** `crates/bumbledb/src/exec/sink.rs:385-388` (`seen: Option<WordMap<()>>`, elided only on the plan proof); `crates/bumbledb/src/exec/sink/aggregate/sink.rs:22-24` (`begin_scan` declines the scan-fold pushdown whenever `seen.is_some()`); sink.rs:204-213 routes to the dedup arm; `crates/bumbledb/src/exec/sink/aggregate/fold_batch.rs:54-62` — one `seen.insert(key)` of the whole slot array per emitted row, survivors then counted arithmetically.
- **Duplicates come from raw suffix iteration.** `crates/bumbledb/src/exec/colt/iter.rs:67-71` — `NodeState::Unforced(_) if is_suffix => self.iter_positions(...)`. This is the Free Join paper's COLT suffix rule verbatim (docs/free-join-paper/arXiv-2301.10841v2/tex/04-optimizations.tex:277-283: "check if it is a suffix of the relation schema: if yes, then we simply iterate over that vector"). r6's node0 cover is a single arity-2 level (suffix ⇒ raw rows, so parallel (src,dst) edges repeat) and occ1's dst level under T1[v1] is likewise raw.
- **The forced map IS the distinct set with an exact O(1) cardinality.** `crates/bumbledb/src/exec/colt/force.rs:7` (`force` builds the keyed map once per node, shared across all parents probing it); `crates/bumbledb/src/exec/colt/count.rs:58-63` (`KeyCount::Exact(map.len)` for forced nodes). Today `key_count` feeds only cover choice (`exec/run/pump.rs:78`) and anti-probes — no count pushdown exists.
- **Distinct paths = distinct bindings.** `crates/bumbledb/src/plan/fj/derive_nodes.rs:29-45` — a cover must bind exactly the node's new vars, so forced-key enumeration at each node yields distinct new-var tuples and the concatenation is a distinct full binding. r6 has no conditions, no filters, no point probes.
- **The volume and the timing.** `crates/bumbledb-bench/src/scenarios/rings/corpus.rs:26-31, 87-101` — 60K transfers + 1-in-8 reciprocal echoes ≈ 67.5K edges, 15% endpoint mass on ~20 hubs ⇒ Σ_v indeg(v)·outdeg(v) ≈ 20·(506)² + tail ≈ 5.3M emitted bindings. `bench-out/night-2026-07-20/scenarios/scenarios.md:59` — r6 ours p50 = 135,264us ⇒ ~25ns/binding, per-row hash-insert scale.

### Bench impact

r6 is the slowest non-DNF rings lane (135.3ms vs 674.2ms SQLite, ratio 0.20 in a scenario whose other lanes run 0.09–0.22 with far more join work). The telescoped plan does: force occ0's cover level once (O(67.5K)), then per distinct (v0,v1) pair (≲60K) one probe of occ1's trie at v1 plus one forced-map `len` read — the [v1]-subtrie forcings amortize to O(67.5K) total across distinct v1. Low single-digit milliseconds is the right order; the seen-set (holding one 3-word entry per distinct 2-path) disappears along with the 5.3M-row emit loop. The same license applies to any single-rule Count-distinct-shaped fold whose emitted variables are all cover-bound with no position (row) enumeration and no membership point probes — the dedup-heavy graph lanes share the shape.

### Suggested fix

Generalize the seen-set elision license from fact-shape (key coverage over `Functionality` statements, provably_distinct.rs) to plan-shape: single-rule AND every emitted variable bound by a cover enumerated as forced distinct keys AND no point-probed position enumeration ⇒ the emitted key stream is duplicate-free by construction, elide `seen`. Execution side: give the plan a way to demand forced iteration at suffix levels (inverting the paper's §4 suffix laziness, which is a false economy exactly when a spanning seen-set re-derives what forcing computes), and for a bare `Count` leaf under this license, fold the forced map's `KeyCount::Exact` len per parent instead of enumerating rows. Residual filters at a level keep the elision valid (rejection cannot mint duplicates) but disable the O(1) len shortcut at that level only.
