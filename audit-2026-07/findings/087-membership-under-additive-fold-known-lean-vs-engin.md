## Membership under an additive fold: the fold-domain divergence is fenced out of the conformance corpus instead of pinned

category: lean-rust-drift | severity: medium | verdict: CONFIRMED | finder: r2:lean-unswept-modules

### Summary

The conformance lane exists because "the engine and the naive model were written from the same docs — a shared misreading passes every two-way differential forever" (crates/bumbledb-bench/src/conformance.rs:4-10). For exactly one query class — a membership binding (element-typed term on an interval field) under a `Count`/`Sum` head — the repo knows the Lean side and the engine side compute different aggregate fold domains, and instead of resolving the divergence (theorem, countermodel, or validation refusal), the corpus builder excludes the class (`Exclusion::AggregateMembership`). The third oracle therefore can never adjudicate membership-under-additive-fold, and the only remaining gate on it is the two-way differential the lane was built to distrust.

### Evidence (all verified in-tree)

- **The fence**: crates/bumbledb-bench/src/conformance.rs:806-814 — `if fresh > var_count && rule.finds.iter().any(additive_fold) { return Err(Exclusion::AggregateMembership); }`; `additive_fold` (conformance.rs:748-756) matches `Count`, `Sum`, and measure-`Sum`. Module doc records it at conformance.rs:51-63; the fence test `membership_under_an_additive_fold_is_fenced` is at conformance.rs:1976-2003. The exclusion is counted in the `Report` (conformance.rs:181, 1182) — recorded, but recording is not adjudication.
- **The Lean fold domain**: lean/Bumbledb/Conformance.lean:465-488 — `varCount` takes the max over `r.body.allVars` (plus head vars), and `ruleBindings` dedupes `fullRow (varCount r)` over the LOWERED rule's states. The serializer's lowering mints a fresh interval variable per membership position (conformance.rs:780-789: `Term::Var(interval_var)` binding + `PointIn` condition), so that fresh variable is inside the Lean glue's fold domain.
- **The engine fold domain**: crates/bumbledb/src/ir/normalize/normalize.rs:147-243 — `is_membership` + `lower_atom` turn a membership position into a filter (`PointIn`-family), never a binding; crates/bumbledb/src/exec/sink.rs:20-26 — "the fold domain of every aggregate is the group's set of distinct full bindings over all query variables". The original query has no interval variable at the membership position, so two stored intervals covering one point that agree on all declared variables are ONE engine binding and TWO Lean rows.
- **The naive model matches the engine**, not Lean: crates/bumbledb-bench/src/naive/query.rs:244, 438, 1148 fold the distinct full bindings of the original rule; membership is read as the typing-rule filter (query.rs:182-220).
- **The proved theorem stops at set semantics**: lean/Bumbledb/Query/Membership.lean:1218-1227 — `membership_lowering_preserves` states `t ∈ surfaceRuleAnswers … ↔ t ∈ ruleAnswers …`, i.e. tuple-membership (set-semantics) preservation only. No theorem in Membership.lean or Aggregates.lean addresses aggregate folds over a lowered membership rule (grep confirms no aggregate-over-membership result exists).
- **No engine-side refusal**: `AggregateMembership` exists only in the bench crate; crates/bumbledb/src/ir/validate has no membership×aggregate interaction, so the engine accepts and executes the shape with its filter semantics, unpinned by any proof or oracle.
- **The doc does pick a side** (correction to the finding as filed): docs/architecture/20-query-ir.md:277 — "Every aggregate folds the group's set of distinct full bindings" of the query — makes the engine/naive reading the normative one, since the original query binds no interval variable at a membership position. The semantics is chosen on paper; it is enforced by nothing machine-checked. The divergence lives in the Lean CONFORMANCE GLUE's decision to fold the lowered rule's `allVars` (Conformance.lean:484-488), which silently changes the fold domain the doc defines — and the fence hides that from the lane rather than surfacing it as a disagreement or fixing the glue.

### Failure scenario

A host counts events overlapping a timestamp: `find x, count :where events[account x, span @ts]` with two stored rows `(acct1, [1,10))` and `(acct1, [3,12))` and `ts = 5`. Engine and naive both answer `(acct1, 1)` (one distinct binding); the Lean glue's reading of the lowered rule answers `(acct1, 2)` (two distinct full rows, the fresh interval variable differing). Because querygen fences the shape out of the corpus, no conformance case ever exercises it, `lake exe conformance` passes forever, and if either the engine's fold domain or the Lean glue drifts in this class, no gate fires. Per the audit lens this is also a representation smell: a semantic question answered by a corpus-builder branch (the fence) rather than by the representation (a Lean form in which the membership fold domain is spelled correctly).

### Suggested fix

The doc already rules (20-query-ir.md § aggregation: fold the ORIGINAL query's distinct bindings — a membership term selects, it does not bind), so make the Lean side say the same thing and lift the fence:

1. In lean/Bumbledb/Conformance.lean, fold over the SURFACE rule's variable width (pre-lowering `varCount`, excluding serializer-minted fresh variables — the serializer can carry the original variable count in the case JSON), so the fresh interval variable is projected away before `dedupRows`; or equivalently dedupe `fullRow` restricted to the original ids.
2. Prove the licensing lemma the module doc says is missing: aggregates over the lowered rule with fresh variables projected out equal aggregates over the surface reading (a `membership_lowering_preserves_fold`-shaped result, or at minimum a checked countermodel documenting why the naive `allVars` fold is wrong).
3. Delete the `AggregateMembership` fence (conformance.rs:806-814) and its test, letting querygen's membership-under-Count/Sum draws into the corpus so the third oracle adjudicates the class on every push.
