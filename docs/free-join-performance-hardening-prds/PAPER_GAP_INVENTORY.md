# Paper Gap Inventory

This inventory tracks known mismatches between Bumbledb, Rosetta, and the local Free Join paper source. Each implementation gap has exactly one target PRD. Product rejections are not implementation work and cite Rosetta's set-first contract.

| id | gap | risk | target_prd | acceptance_signal | status |
| --- | --- | --- | --- | --- | --- |
| G01 | Paper uses bag semantics in examples, while Bumbledb must keep set semantics. | Copying paper bag behavior would violate duplicate-free fact and projection semantics. | product-rejected | Rosetta remains explicit that base relations, solution bindings, and projected results are sets. | rejected |
| G02 | Paper assumes full conjunctive queries internally, while Bumbledb exposes projection. | Projection could accidentally multiply witness duplicates or hide missing full-binding work. | 14 | Encoded set sinks consume complete bindings and deduplicate projected facts before decode. | open |
| G03 | Paper assumes selections are pushed to base tables, while Bumbledb applies source filters after relation image columns are available. | Selective and empty JOB queries pay broad scan and load cost before proving emptiness. | 12 | Source filter traces show rows tested, survivor counts, and zero-source short-circuiting before COLT map work. | open |
| G04 | Paper `GHT::iter` returns an iterator, while Bumbledb returns `Vec<EncodedTuple>`. | Eager materialization allocates heavily and weakens COLT laziness. | 10 | No hot-path GHT method returns `Vec<EncodedTuple>`. | open |
| G05 | Paper `iter_batch` is true batching, while Bumbledb chunks a fully materialized vector. | Vectorized execution cannot reduce allocation or improve locality if all tuples are already materialized. | 10 | Batch tests prove bounded batch production without first collecting every tuple. | open |
| G06 | Paper COLT stores offsets into column-oriented base data, while Bumbledb columns store per-cell `Vec<u8>` in `Vec<Vec<u8>>`. | Per-cell allocations damage scans, filtering, tuple construction, and future NEON kernels. | 09 | Base-image columns are contiguous fixed-width buffers with zero-copy `value_at`. | open |
| G07 | Paper lazily builds tries only when needed, while Bumbledb builds source offset lists and may force maps through cover choice and iteration. | Empty or highly selective branches still allocate and scan structures they never need. | 11 | COLT traces prove suffix iteration avoids force and `get` forces only needed nodes. | open |
| G08 | Paper starts from optimized binary plans, while Bumbledb uses deterministic generated candidates and simple scores. | The current planner cannot fully explore the Free Join design space or reuse optimizer-quality cost data. | 16 | Dynamic cover and plan scoring expose exact or labeled-estimated costs with traceable decisions. | open |
| G09 | Paper factors lookups conservatively, while Bumbledb has factoring without traceable rewrite evidence. | Plan rewrites may be hard to audit when performance changes. | 05 | `generate_plan_candidates_with_trace` emits factorization spans with source node, destination node, outcome, and reason. | closed |
| G10 | Paper dynamic cover selection chooses the source with fewest keys, while Bumbledb often uses offset counts as estimates. | Skewed sources can choose bad covers and perform unnecessary probes. | 16 | Cover-choice trace distinguishes exact key counts, estimates, and tie-breaks. | open |
| G11 | Paper emphasizes vectorized execution, while Bumbledb public default is scalar. | The default path misses a core paper optimization and obscures batch-locality opportunities. | 15 | AArch64 NEON vectorized mode is real batching, measured, and eligible for default only after gates pass. | open |
| G12 | Paper implementation is main-memory only, while Bumbledb adapts Free Join to LMDB snapshots. | Paper assumptions about random access and build cost can mislead storage decisions. | 18 | Storage v6 stats and accelerators are documented as LMDB snapshot adaptations and correctness-optional. | open |
| G13 | Paper discusses materialization bottlenecks, while Bumbledb still decodes and materializes in sink paths that can dominate small result queries. | Duplicate witnesses can multiply decode and allocation cost despite set output. | 14 | Decode count equals final projected cell count, not logical witness count. | open |
| G14 | Paper does not address durable storage accelerators, while Bumbledb has namespaces but no real value accelerator path. | Adding accelerators without discipline could make correctness depend on optional structures. | 18 | Queries produce identical exact results with accelerators enabled and disabled. | open |
| G15 | Current explain says timings and allocations are not collected. | Diagnostics misrepresent the hardened engine and leave benchmark regressions opaque. | 04 | `ReadTxn::execute_query_profiled` emits canonical trace spans, explain points to profiled execution traces, and PRD 06 harvest records real JOB trace files. | closed |
| G16 | x86 SIMD is forbidden for this suite; vectorization must be AArch64 NEON only. | Adding x86 SIMD would expand support burden and violate the user's NEON-only direction. | product-rejected | The x86 forbidden search returns no matches under `crates/`, and any SIMD implementation is AArch64-gated. | rejected |

## Update Rule

When a PRD closes a gap, update its `status` to `closed` and replace the `acceptance_signal` with the concrete test, benchmark, or trace evidence. If a later PRD changes ownership, update `target_prd` in the same commit that changes the ordered suite.
