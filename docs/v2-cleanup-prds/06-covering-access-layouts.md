# PRD 06: Covering Access Layouts

## Goal

Replace primary-key-plus-secondary-identity layout generation with full covering tuple access paths.

Every generated or explicit access path must encode every relation field exactly once. The leading fields provide the access prefix. All remaining fields are appended in canonical schema field order.

This is the architectural pivot that makes set semantics simple and lets storage delete the current-row payload store in PRD 07.

## Current State

Current index generation in `crates/bumbledb-core/src/schema.rs:943-1068`:

- Always creates a `primary` index from `relation.primary_key.fields`.
- Auto-generates `IndexKind::Ref` indexes for `ValueType::Ref` fields.
- Adds range indexes for `field.indexing.range`.
- Adds unique and FK generated indexes.
- Appends primary key fields to non-primary indexes as `ComponentRole::Identity`.
- Only primary index is guaranteed to cover full row.

## Target Model

An access path is a full tuple permutation.

For a relation with fields:

```text
[id, holder, currency]
```

The canonical covering unique `by_id` has components:

```text
[id, holder, currency]
```

The unique `holder_currency` has components:

```text
[holder, currency, id]
```

The FK `holder_fk` has components:

```text
[holder, id, currency]
```

The range index `by_created_at` for `[created_at]` has components:

```text
[created_at, id, other_fields...]
```

Every access key fully decodes to the row.

## Required IndexKind

Replace:

```rust
pub enum IndexKind {
    Primary,
    Ref,
    Unique,
    ForeignKey,
    Range,
    Equality,
    Permutation,
}
```

with:

```rust
pub enum IndexKind {
    Covering,
    Unique,
    ForeignKey,
    Range,
    Equality,
    Permutation,
}
```

`Covering` means the access path generated by the relation's one `Unique { covering: true }` constraint.

`Unique` means a non-covering unique constraint from the logical perspective, but its physical access path still covers the full tuple by appending remaining fields.

## Required ComponentRole

Replace `ComponentRole` with:

```rust
pub enum ComponentRole {
    Leading,
    Covering,
}
```

Delete `Identity`. The old role existed only because secondary indexes appended primary keys.

## Required Layout Generation

In `RelationDescriptor::index_candidates`, generate candidates in deterministic order:

1. The one covering unique constraint as index name `covering` or `unique_{name}`.
2. Non-covering unique constraints as `unique_{name}`.
3. Foreign keys as `by_fk_{name}`.
4. Range-indexed fields as `by_{field}`.
5. Explicit indexes.

Recommended candidate type:

```rust
struct IndexCandidate {
    name: String,
    kind: IndexKind,
    fields: Vec<String>,
    unique_constraint: Option<String>,
}
```

If the covering unique constraint is named `by_id`, the physical access path name should be deterministic. Two acceptable options:

- Always call it `covering`.
- Call it `unique_by_id`.

Prefer `covering` for architectural clarity, but this requires updating many tests and query-image references currently hard-coded to `primary`.

Do not preserve `primary` as an alias.

## Required Component Construction

Replace old `index_components` logic with:

```rust
fn index_components(
    &self,
    index_name: &str,
    leading_fields: &[String],
) -> Result<Vec<IndexComponent>> {
    let mut components = Vec::with_capacity(self.fields.len());
    let mut seen = BTreeSet::new();

    for field_name in leading_fields {
        let field = self.field(field_name).ok_or_else(|| SchemaError::UnknownField {
            relation: self.name.clone(),
            field: field_name.clone(),
        })?;
        if !seen.insert(field.name.clone()) {
            return Err(SchemaError::DuplicateIndexField {
                relation: self.name.clone(),
                index: index_name.to_owned(),
                field: field.name.clone(),
            });
        }
        components.push(IndexComponent::new(field, ComponentRole::Leading));
    }

    for field in &self.fields {
        if seen.insert(field.name.clone()) {
            components.push(IndexComponent::new(field, ComponentRole::Covering));
        }
    }

    Ok(components)
}
```

This guarantees every access path covers the full row.

## Key Size Policy

Full covering access paths increase key sizes. This is acceptable.

If any generated key exceeds LMDB max key size, schema validation must fail with `KeyLayoutTooLarge`.

Do not add fallback current-row payload behavior for large rows. That would preserve the old branchy model.

## Query Image Impact

Current query image fallback scans `"primary"` at `crates/bumbledb-lmdb/src/query_image.rs:1458-1486`.

Update it to scan the covering access path name.

Recommended helper on `StorageSchema`:

```rust
pub(crate) fn covering_layout(&self, relation: &str) -> Option<&CurrentIndexLayout>;
pub(crate) fn covering_index_name(&self, relation: &str) -> Option<&str>;
```

Avoid hard-coding `"covering"` throughout storage/query. Centralize lookup.

## Tests Required

- Every generated layout contains every relation field exactly once.
- Covering unique layout is `IndexKind::Covering`.
- Non-covering unique layout has unique fields first and all remaining fields appended.
- FK layout has FK fields first and all remaining fields appended.
- Range layout has range field first and all remaining fields appended.
- Explicit permutation layout respects explicit leading order and appends remaining fields.
- No generated `IndexKind::Ref` exists.
- No generated `IndexKind::Primary` exists.
- Key-size validation still fires for large covering layouts.
