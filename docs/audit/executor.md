## Verdict

I found **no correctness bug** in the executor, the kernels, or the guard-probe dispatch: every concrete scenario I traced — cover choice per node entry, batch draw/resume tokens, two-phase probing, survivor compaction, residual word semantics on biased I64 words, D2 SkipSuffix unwinding at every depth, aggregate fold domains, selection-level cursor starts, and guard-vs-join result equivalence — produces the same answers as the paper's algorithm under the documented set-semantics deviations. The historical cover-rebind hazard is structurally excluded (covers are exactly-the-new-vars at plan validation, and the runtime cover is chosen once per node entry, before any batch is drawn, so it cannot differ between batches). The findings below are two doc-level contradictions (the code follows the newer ruling in both cases), plus defense-in-depth and envelope-edge notes; none produces wrong results or UB on valid input today.

## Findings

### [MEDIUM] 30-execution.md still states the superseded cover-choice rule

`docs/architecture/30-execution.md:74` vs `crates/bumbledb/src/exec/run.rs:554`.

The invariant at stake: docs describe the system in the present tense (README rule 5); the §4.4 bullet is normative text. Line 74 says *"v0 rule: prefer the smallest `Exact`; otherwise smallest `Estimate`"* — the exact label-first rule the perf-suite section of the same document (lines 272–276, "Magnitude-first cover choice") repudiates as *"the measured balance wrong-cover"*, and which `better_cover` (run.rs:554–560) correctly does **not** implement: it compares magnitudes first and uses the label only on ties. Concrete failure scenario: not a runtime failure — the code is right — but a reader (or future implementer) taking line 74 as normative would reintroduce the measured 500-key-forced-map-over-7-row-view regression, and the two paragraphs of one doc currently give contradictory instructions. Fix direction: amend the §4.4 bullet to state magnitude-first (or point it at the perf-suite paragraph) in place.

### [MEDIUM] D4 says NEON handles survivor compaction; compaction is scalar on every target

`docs/architecture/30-execution.md:176` vs `crates/bumbledb/src/exec/kernel.rs:57-65` (and module docs kernel.rs:13-16).

The invariant: same present-tense rule. Line 176 states NEON *"handles fixed-width predicate scans **and survivor compaction** and nothing else"*; `compact_u32_by_mask` is the scalar cursor-write on every target, and kernel.rs's module docs explain why (no NEON compress instruction; that is SVE, which Apple Silicon lacks) while citing the doc's *other* sentence (line 171, "NEON compress **or scalar cursor-write**") as sanction. The two doc sentences disagree with each other and the first disagrees with the code. No runtime consequence — the scalar compaction is correct (verified below) — but the doc misdescribes the shipped kernels. Fix direction: amend line 176 to "fixed-width predicate scans" only.

### [NOTE] Aggregate skip-legality rests on a single point of enforcement

`crates/bumbledb/src/exec/run.rs:504-515`, `crates/bumbledb/src/exec/sink.rs:231-303`, `crates/bumbledb/src/plan/fj.rs:312`, `crates/bumbledb/src/api/prepared.rs:352`.

The documented invariant: D2's skip is *never legal under an aggregate sink*. Verified: `AggregateSink::emit` returns `Flow::Continue` at both of its exits (sink.rs:240, 302), so no skip can originate under aggregation today. However, the executor's gating bits (`sink_relevant`) are computed from `witness.group_key()` — the non-aggregated find variables — for aggregate plans too (prepared.rs:352 → fj.rs:312). If any future change made the aggregate sink (or any new sink) return `SkipSuffix`, the executor would honor those bits and skip suffixes binding aggregated-over and existential variables — silently wrong sums/counts, with no assertion to catch it. Concrete scenario requires a future code change, hence NOTE, not a bug. Fix direction: either force `sink_relevant = true` on every node of an aggregate plan (making the bits themselves encode illegality) or `debug_assert!` in `run_node` that a `SkipSuffix` never arrives when the plan was built for an aggregate shape.

### [NOTE] Phase 1 hashes (and counts) probes that will never use the hash

`crates/bumbledb/src/exec/run.rs:410-426`, `crates/bumbledb/src/exec/colt.rs:353-360`.

When a sibling's cursor is a pinned `Cursor::Row`, `get_prehashed` resolves the probe by `position_matches` and ignores the hash entirely — but phase 1 still computes `hash_key` for every survivor and fires `counters.probe_hash`. Pure ALU waste on FK-walk-shaped deep nodes (where pinned rows dominate), and EXPLAIN's per-subatom `hashes` statistic counts hashes that never gated a bucket load, which can mislead a reader correlating hashes to probes. No correctness impact. Fix direction: branch phase 1 on `matches!(s_cursor, Cursor::Row(_))` per sibling (one branch per sibling per batch, not per tuple), or document the statistic as "keys gathered", not "hashes used".

### [LOW] u32 conversion expects panic beyond the stated envelope rather than degrading

- `crates/bumbledb/src/exec/kernel.rs:94,123,151,182` — filter kernels store positions as u32 via `try_from(...).expect`; a column with ≥ 2³² rows panics mid-scan.
- `crates/bumbledb/src/api/prepared.rs:221-224` — `ResolveMemo` packs result-buffer byte offsets into u32; a result byte-heap crossing 4 GB (huge distinct-string output) panics during finalize.

Both sit orders of magnitude above the ≤10⁷-fact scale axiom, and "resource limits: none in v0, the OS is the backstop" (30-execution) arguably sanctions the panic as the backstop's cousin. Not reachable in the design envelope; recorded so the boundary is written down. Fix direction: none needed for v0; if the envelope ever grows, these are the two width cliffs in the read path.

## Checked and sound

- **Join loop vs paper Fig. 5 / §4.3 / §4.4.** Per node entry: cover chosen among the plan's cover set by labeled key count (run.rs:326, 525-541), cover iterated in batches at the occurrence's current `(cursor, level)`, every other subatom probed two-phase, residuals compacted, recursion per surviving entry, journal-restored unwind. Verified against the paper's generic loop, factorized loop, and vectorized loop; the randomized differential test (60 cases × 3 shapes × random orders × batch {1,7,128}) corroborates.
- **Cover-rebind hazard, both forms, excluded.** (a) Mixed-var subatoms (carrying an already-bound var) are never covers: `derive_nodes` (fj.rs:497-505) admits only subatoms whose vars are *exactly* the node's new vars, per the documented deviation; regression test `covers_never_rebind_an_already_bound_variable` pins it. (b) The dynamic cover cannot differ between batches of one node entry: `choose_cover` runs once, before the `'outer` batch loop, and `sources`/`residual_sources` are recomputed at each node entry against that choice (run.rs:326-365). `Source::Batch(i)` therefore always reads a *new* variable from the cover's key column `i`, whose order matches the trie-schema level order and the slot map (verified index-by-index).
- **Cursor/level hygiene.** `execute` rebuilds `cursors` from `colt.start()` per execution (run.rs:297-303) — the post-selection cursor for every occurrence, cover and non-cover alike; all public COLT entry points (`iter_batch`, `get_prehashed`, `ensure_forced`) offset by `selection_levels`, so join level 0 means "below the selections" everywhere. The per-batch journal saves node-entry `(cursor, level)` once per batch and restores after every entry's recursion (before the Flow check, so the SkipSuffix `break 'outer` also unwinds restored); across batches the saved values are re-read after restoration and hence identical. No state survives across executions except deliberately memoized forced tries (sound under generational immutability).
- **Batching.** Buffer sizing `batch × max_subatom_arity` dominates every level's arity (the cover's iter, each sibling's probe keys); `iter_batch`'s undersized-buffer assert can't fire from the executor. Partial batches, empty relations, batch > row count, batch = 1 (the degenerate scalar case), and 1024 all verified equal by test; the resume tokens (`Root` index / packed chunk+offset with a distinct EXHAUSTED encoding / dense map index / Row one-shot) cannot return a spurious zero mid-stream and cannot be misread across state changes, because a node being iterated at one node entry cannot be forced by the recursion beneath it (descendants only touch child cursors, sibling occurrences own separate `Colt`s, and an occurrence appears at most once per node).
- **Stale-scratch safety.** `sibling_children[s][e]` is read only for entries in the final survivor list, all of which were written during this batch's probe of every sibling; entries eliminated early are never read. `survivors`/`mask`/`hashes` are cleared per batch/per sibling; `entry_keys` tails beyond `yielded` are never indexed.
- **D2 SkipSuffix, at every unwind depth.** `ProjectionSink` signals on every emit; by induction a `SkipSuffix` reaching node N implies every node > N has `sink_relevant == false`, and node N breaks only when its own bit is false — so the entire skipped suffix binds exclusively non-projected variables and the emitted projected fact is fully determined by nodes < N. Absorption at the first `sink_relevant` node resets flow to `Continue` and resumes the same batch (token intact). `sink_relevant` = node's new vars ∩ `group_key` (for projection queries, the full find-var set — verified in `ir/validate.rs:128-135`). Duplicate-emitting shapes (gate atoms, suffix-level duplicate keys) collapse in the sink and are curtailed by the skip. The aggregate sink never emits `SkipSuffix` (see NOTE above), and the executor honors `Flow` identically at every depth (leaf return, per-entry check, per-node propagation).
- **Residuals.** Placed at the earliest node binding both sides (fj.rs:316-332); evaluated per batch, after all sibling probes, for every survivor — across batches too, since the mask/compaction runs inside the `'outer` loop. Slot sources are ancestor-bound (stable during the node's batches — descendants only write their own new vars); batch sources are the node's new vars, present in any admissible cover. Word semantics: image words are `u64::from_be_bytes(canonical)`, so I64 words are sign-flipped-biased and unsigned word order **is** numeric order (`encoding.rs:112-119`); `CmpOp::compare` on `u64` is therefore value-faithful for U64/I64, and `Eq`/`Ne` on intern ids and widened bytes are injective. Ordered residuals on String/Bytes/Bool/Enum are unreachable: `cmp_legal` (validate.rs:87-94) rejects `Lt/Le/Gt/Ge` for every non-integer type at the IR boundary, for var-var comparisons included.
- **Set-semantics elision (`distinct_bindings`).** Verified airtight: if every occurrence's bound fields (var-bound or Eq-constant) cover a unique constraint, a binding determines each occurrence's fact uniquely, positions appear exactly once in a trie, and a zero-var occurrence with covering Eq selections yields at most one position — so no duplicate binding can reach the aggregate sink with the seen-set elided. Gate atoms without coverage correctly leave the flag false.
- **kernel.rs.** `compact_u32_by_mask`: in-place cursor-write with `write ≤ read` invariant — correct for empty/full/mixed masks (tested). NEON `filter_eq_u64`/`filter_range_u64` (2-lane) and `filter_eq_u8` (16-lane): the branchless `out[write] = pos; write += cond` pattern stays in bounds because `write ≤ start + matches-so-far ≤ start + index < start + len` after the `resize`; junk at `out[write]` is overwritten or truncated. Tail loops handle remainders of 1 (u64) and 1–15 (u8); lengths 0/1/2/3/15/16/17/31/32/33/100/257 are property-tested bit-identical against the scalar reference, with extremes (0, u64::MAX) and inverted ranges (lo > hi → empty, matching `RangeInclusive::contains`). `vld1q` on slice-interior pointers is sound (element-aligned; A64 permits unaligned vector loads); no write goes through a raw pointer except the 16-byte mask store into a stack array. Range decomposition in `view.rs::kernel_scan` (Lt→`0..=c-1` with the c=0 empty case, Gt→`c+1..=MAX` with the c=MAX empty case) is exact on biased words.
- **Guard-probe dispatch.** Classification (single occurrence, no residuals, Eq-constant coverage of a unique constraint or the full fact) is sound: uniqueness guarantees the join over the equivalent filtered view yields at most the one fact the `U`/`M` probe returns; remaining filters (extra Eq — including a second, contradictory Eq on a key field — ranges, `Ne`, `FieldsCompare`) are evaluated on the fetched fact with the same word encodings as image filters; guard-key bytes are canonical field encodings in constraint-field order, matching commit's `U`-key derivation (`storage/commit.rs:537`); the membership fallback concatenates all fields in declaration order = `fact_bytes`, and degenerates correctly to the empty fact for nullary relations (vacuous `all`). Param and `PendingIntern` misses resolve to the never-minted sentinel (u64::MAX, mint-asserted) — `Eq` misses, `Ne` matches everything, identically on both paths; the miss never interns and never errors (tested). Aggregates over guard results fold the ≤1 binding through the ordinary sink with the (trivially true) distinct flag; guard/FreeJoin equivalence is asserted by test. No image, view, or cache is reachable from `execute_guard` — the cold-database property holds by API shape.
- **Counters seam.** `NoopCounters` is zero-sized with empty inlined methods; `CountingCounters` methods are plain array increments; neither can affect results, and the phase-1-before-phase-2 ordering claim is pinned by `phase_one_hashes_the_whole_batch_before_any_phase_two_probe`.
- **Allocation discipline (spot-checked, not gate-run).** Warm `run_node` paths write only into retained-capacity buffers (`survivors`/`hashes`/`mask` bounded by `batch`, `journal`/`sources` reaching fixpoint after the first execution); `set_batch_size` reallocates outside any measured window by construction.
