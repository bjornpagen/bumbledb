## Objective

Re-true every measured claim on the final tree. The layer-law wins that landed
(W1 fixpoint, W3 finalize, W4, W8's probe sort) and any C1 verdict have MOVED the
numbers, so the five README charts and every numeric claim are stale. Regenerate
them honestly under the measurement protocol so the `1.0.0` README tells the truth.

## Context

- The bench binary refuses to time against an unverified build (per-binary
  verification stamp); `bench` needs a fresh `verify` stamp first. Measurement law:
  `scripts/measure.sh` mutex, interleaved same-session A/B for any comparison,

  `assets/*.svg` regenerated by `scripts/bench_viz.py` over min-of-3 runs.
- This is Phase D of the old `TODO.md`, now unblocked by the landed wins and the
  quiet machine.

## Work

1. **Verify + stamp.** `bumbledb-bench gen` (digest-keyed corpus cache) then
   `bumbledb-bench verify` — the 2,862-case two-oracle gate (SQLite parity + naive
   model) — producing the fresh per-binary stamp. A red oracle stops everything
   (it is a correctness regression, not a chart issue).
2. **Time.** Through `scripts/measure.sh`: 3 durable `bench` runs + 3 ephemeral
   runs + `scenarios`, per the README's own reproduction recipe. Fresh data per
   rep (TAGE memorization); min-of-3 across processes; contaminated blocks excluded
   and counted.
3. **Regenerate the charts.** `scripts/bench_viz.py` over the three runs +
   scenarios → all five `assets/*.svg` (`bench-vs-sqlite`, `speedup`, `tails`,
   `writes`, `scenarios`). Confirm ALL-WIN holds (every gated read family beats
   SQLite p50) — if a family LOST, that is a stop-the-line regression, reported,
   not charted around.
4. **True the README.** Update every numeric claim in `README.md` to the new
   measured values (the geomeans, the per-family multiples, the write-curve band,
   the p99 figures) and any surviving normative number elsewhere (e.g. the
   ephemeral-store band). Numbers are EXPECTED to move up (W1–W3 are collectively
   bigger than the whole layer-law campaign) — re-true, do not just re-earn.
5. Record the run (the campaign-discipline: the numbers + the machine conditions)
   in the commit body; the charts' provenance is the three run dirs under
   `bench-out/`.

## Technical direction

- Absolute numbers under co-tenancy are VOID — the machine must be idle (this is
  Wave 3's gate). State the tier (L1/L2/DRAM/displaced) of every headline number.
- Do NOT touch engine code here — this is measurement + README truing only. A
  number that reveals a regression is a finding for a fix PRD, not something to
  massage.
- ALL-WIN is the gate for the read charts; the write chart is reported honestly
  (fsync-bound, bulk favors SQLite) — publish it anyway, as the README already does.

## Passing criteria

- Fresh `verify` stamp on the final binary (2,862 cases green); `bench` refuses
  without it — so a stamped run is proof the oracles passed.
- Three durable + three ephemeral runs + scenarios completed through `measure.sh`;
  ALL-WIN holds; contaminated blocks excluded and counted.
- All five `assets/*.svg` regenerated from the runs; `README.md` numbers trued to
  the measured values; no stale figure remains (grep the README against the run
  reports).
- Commit in the repo's voice (numbers + conditions in the body); push. (The
  version bump + tag are R2, the owner's.)
